# 系统监测与诊断设计

本文说明当前边端 `SystemMonitor` 的已落地能力。早期把系统监测作为新增项的设计已经收敛，当前生产交付中系统监测、配置拉取/写回、远程诊断、实时点位快照和网口直连维护都由 `SystemMonitor` 承担。

## 1. 当前定位

`SystemMonitor` 是边端系统级维护服务，不属于某个采集协议驱动。它和 `MqttDriver`、`ModbusRtu`、`Dlt645Driver`、`DioDriver`、`CanDriver`、`EventEngine` 并列运行。

核心职责：

- 采集 CPU、内存、磁盘、负载、进程和 4G 状态。
- 根据平台订阅租约上报系统资源。
- 资源超过阈值时主动上报告警。
- 执行白名单远程诊断命令。
- 通过 MQTT 拉取边端配置快照、写回配置文件和执行文件操作。
- 通过 `edge/system/monitor/points` 发布实时点位快照，供平台和 Win 工具监听。
- 内嵌网口直连维护 HTTP API，用于 Win 工具在本地直连场景下维护设备。

不属于 `SystemMonitor` 的职责：

- 协议采集。
- 告警和变位规则判断。
- 普通业务 telemetry 全量上传。
- EMS 策略执行。

## 2. 服务启动

生产使用 systemd 模板：

```bash
systemctl start system-monitor@monitor-service.service
systemctl status system-monitor@monitor-service.service
```

实际命令：

```bash
/opt/modbus-gateway/bin/SystemMonitor \
  --app-config /opt/modbus-gateway/config/runtime/apps/monitor-service.json
```

`gateway-services.sh` 会在 `monitor-service.json:systemMonitor.enabled=true` 时启动该服务。

直连维护 API 已融合在 `SystemMonitor` 内部，不再部署独立 `DirectAgent`。如果需要临时关闭直连维护，可手工启动时追加：

```bash
SystemMonitor --app-config /opt/modbus-gateway/config/runtime/apps/monitor-service.json --no-direct-maintenance
```

## 3. 配置文件

主配置：

```text
config/runtime/apps/monitor-service.json
```

关键字段：

| 字段 | 当前含义 |
| --- | --- |
| `systemMonitor.enabled` | 是否启动系统监测服务 |
| `systemMonitor.defaultIntervalMs` | 无特殊订阅时的默认采样周期 |
| `systemMonitor.minIntervalMs` | 平台订阅允许的最小采样周期 |
| `systemMonitor.subscriptionTtlSec` | 订阅租约 TTL |
| `systemMonitor.configPullChunkBytes` | 配置快照 MQTT 分片大小，生产默认 `131072`，代码上限 `256 KiB` |
| `systemMonitor.allowedCommands[]` | 远程诊断白名单命令 |
| `systemMonitor.realtimeMeterLeaseFile` | 实时监听表计租约文件，供 `MqttDriver` 控制 realtime 上报范围 |
| `systemMonitor.cellular.*` | 4G 状态采集配置 |
| `systemMonitor.directMaintenance.*` | 网口直连维护 API 配置 |

当前直连维护默认监听固定维护口：

```json
{
  "directMaintenance": {
    "enabled": true,
    "listenHost": "192.168.1.250",
    "listenHosts": ["192.168.1.250"],
    "listenPort": 9443,
    "allowedClientCidrs": []
  }
}
```

`allowedClientCidrs=[]` 表示不限制客户端 IP；访问仍需要维护密码。当前 Win 工具和边端固定维护密码为 `Kydl2026`。

## 4. MQTT Topic

配置里保存的是基础 topic，运行时按 `machineCode` 追加后缀。以 `COMM202600102` 为例：

| 能力 | 基础 topic | 实际 topic 示例 |
| --- | --- | --- |
| 系统监测订阅 | `edge/system/monitor/request` | `edge/system/monitor/request/COMM202600102` |
| 系统监测回复 | `edge/system/monitor/reply` | `edge/system/monitor/reply/COMM202600102` |
| 系统资源遥测 | `edge/system/monitor/telemetry` | `edge/system/monitor/telemetry/COMM202600102` |
| 系统资源告警 | `edge/system/monitor/alert` | `edge/system/monitor/alert/COMM202600102` |
| 实时点位快照 | `edge/system/monitor/points` | `edge/system/monitor/points/COMM202600102` |
| 诊断请求 | `edge/system/diag/request` | `edge/system/diag/request/COMM202600102` |
| 诊断回复 | `edge/system/diag/reply` | `edge/system/diag/reply/COMM202600102` |
| 配置拉取请求 | `edge/config/pull/request` | `edge/config/pull/request/COMM202600102` |
| 配置拉取回复 | `edge/config/pull/reply` | `edge/config/pull/reply/COMM202600102` |

控制、配置写回和诊断属于控制类通道，平台和 Win 工具应使用 `mqtt.controlQos`，当前默认按 QoS 2 处理；普通遥测继续使用 `mqtt.qos`。

## 5. 系统资源监测

采集内容：

- CPU 使用率。
- 内存使用率。
- 根分区磁盘使用率。
- 负载均值。
- 进程数量。
- 关键服务状态。
- 4G 模块状态。

资源采集优先读系统文件，避免周期性调用重命令：

- CPU：`/proc/stat`
- 内存：`/proc/meminfo`
- 负载：`/proc/loadavg`
- 进程数：`/proc`
- 磁盘：`statvfs("/")`

4G 状态采集顺序：

1. 优先探测 `/dev/ttyUSB2`、`/dev/ttyUSB1`、`/dev/ttyUSB0`、`/dev/ttyUSB3`。
2. 兜底匹配 `/dev/ttyUSB*` 和 `/dev/cdc-wdm*`。
3. 如果存在 `mmcli`，补充 ModemManager 输出。
4. 如果没有 `mmcli`，使用 AT 指令采集 `CPIN/CSQ/COPS/CREG/CEREG/CGATT/CGPADDR/CGSN/CCID`。
5. 网络接口、IP、路由和 DNS 从 `/sys/class/net`、`ip addr`、`/proc/net/route`、`/etc/resolv.conf` 获取。

资源遥测样例：

```json

  "type": "system-monitor",
  "machineCode": "COMM202600102",
  "cpuUsage": 12.4,
  "memUsage": 36.8,
  "diskUsage": 41.2,
  "load1": 0.52,
  "processCount": 113,
  "cellular": {
    "enabled": true,
    "present": true,
    "registered": true,
    "connected": true,
    "simReady": true,
    "operator": "China Mobile",
    "accessTech": "LTE",
    "interfaceName": "ppp0",
    "ipAddress": "10.23.1.8",
    "signalPercent": 76.5,
    "rssiDbm": -67,
    "imei": "863859088879627",
    "iccid": "89860869102570293945",
    "ts": 1780000000000
  },
  "services": [
    {"name": "MqttDriver", "running": true},
    {"name": "EventEngine", "running": true},

  ],

```

## 6. 订阅租约

平台打开系统监测页时发送订阅请求：

```json
{
  "action": "subscribe",
  "machineCode": "COMM202600102",
  "sessionId": "MONITOR_20260626_001",
  "intervalMs": 1000,
  "ttlSec": 30
}
```

边端行为：

- 建立监测租约。
- 租约有效期内按 `max(intervalMs, minIntervalMs)` 上报资源。
- 页面关闭时可发送 `unsubscribe`。
- 如果平台未续租，边端在 TTL 到期后自动停止高频上报。

资源阈值告警不依赖页面订阅。CPU、内存、磁盘和 4G 异常达到阈值时，边端会按 `alertRepeatIntervalSec` 节流上报。

## 7. 实时点位快照

`SystemMonitor` 可以通过共享内存 Router 读取当前点位，并发布到：

```text
edge/system/monitor/points/{machineCode}
```

该 topic 主要服务于平台实时监测和 Win 工具的“打开实时监控后才监听”的场景。它不替代 `MqttDriver` 的业务全量上传。

点位快照支持分片，payload 类型仍为 `snapshot`：

```json
{
  "type": "snapshot",
  "machineCode": "COMM202600102",
  "chunked": true,
  "chunkId": "1780000000000",
  "chunkIndex": 1,
  "chunkCount": 3,
  "meters": [
    {
      "meterCode": "PCS_001",
      "values": [
        {
          "index": 110001,
          "pointCode": "active_power",
          "name": "PCS有功功率",
          "value": 12.3,
          "quality": 1,
          "ts": 1780000000000
        }
      ]
    }
  ]
}
```

实时监听表计租约写入：

```text
/opt/modbus-gateway/run/realtime-meter-leases.json
```

`MqttDriver` 读取该租约后，只在存在实时监控租约时按实时通道高频扫描；没有实时监控时，共享内存扫描压力主要跟随全量上传周期。

## 8. 配置拉取与写回

配置拉取请求通过：

```text
edge/config/pull/request/{machineCode}
```

配置回复通过：

```text
edge/config/pull/reply/{machineCode}
```

当前能力：

- 拉取 `runtime/apps`、`runtime/devices`、`device_identity.json` 等运行配置。
- 大配置按 `systemMonitor.configPullChunkBytes` 分片，生产默认 `128 KiB`。
- 回复中带 `requestId`、`chunkIndex`、`chunkCount`、文件路径、字节数和内容。
- 平台和 Win 工具会按 `requestId` 聚合分片，并显示进度。

配置写回支持：

- dry-run 校验。
- 分片上传大文件。
- 写入允许范围内的运行配置文件。
- 写回完成后按需要重启 `gateway-services.service` 或相关服务。

安全边界：

- 只能操作配置白名单内的路径。
- 不允许通过配置写回写任意系统文件。
- 写回前应由平台或 Win 工具做 JSON 结构、index 冲突、通道引用和驱动启用关系校验。

## 9. 远程诊断

远程诊断只支持白名单命令，不允许平台下发任意 shell。

当前白名单：

| 命令 | 用途 |
| --- | --- |
| `uptime` | 查看系统运行时间和负载 |
| `free_m` | 查看内存 |
| `df_root` | 查看根分区 |
| `top_once` | 单次 CPU/进程快照 |
| `ps_gateway` | 查看网关相关进程 |
| `journal_tail` | 查看最近日志 |
| `systemctl_status` | 查看允许服务状态 |
| `cellular_status` | 查看 4G 状态 |

请求样例：

```json
{
  "cmdId": "DIAG_20260626_001",
  "machineCode": "COMM202600102",
  "command": "df_root",
  "args": {},
  "operator": "admin",
  "ts": 1780000000000
}
```

回复样例：

```json
{
  "cmdId": "DIAG_20260626_001",
  "machineCode": "COMM202600102",
  "success": true,
  "command": "df_root",
  "exitCode": 0,
  "stdout": "Filesystem Size Used Avail Use% Mounted on ...",
  "stderr": "",
  "ts": 1780000000500
}
```

## 10. 网口直连维护

直连维护用于 Win 工具和设备网口直连的现场维护场景。当前实现已嵌入 `SystemMonitor`，不是独立服务。

默认配置口：

```text
ens0: 192.168.1.250/24
```

设备其它出厂网口规划：

```text
ens1: DHCP
ens2: 192.168.3.250/24
ens3: 192.168.4.250/24
```

直连维护 API 提供：

- 健康检查。
- 固定密码认证。
- 配置快照读取。
- 配置 dry-run 和写回。
- 实时点位读取。
- 全量点位快照读取。
- 白名单诊断。
- 批量控制写入。
- 配置包和整包 OTA 任务受理。

维护模式安全边界：

- 默认密码为 `Kydl2026`。
- `allowedClientCidrs=[]` 时允许所有客户端 IP 调用，适合公司内部调试；生产客户交付时可按现场网段收紧。
- API 仍校验维护密码。
- 文件操作仍受配置路径白名单限制。
- OTA 和配置写回成功后由边端统一执行服务重启。

## 11. 平台接口

Java 平台当前接口：

```http
POST /api/system-monitor/subscribe
POST /api/system-monitor/unsubscribe
POST /api/system-monitor/config/pull
GET  /api/system-monitor/config/snapshot/latest
GET  /api/system-monitor/config/pull/progress
GET  /api/system-monitor/history
GET  /api/system-monitor/alerts
POST /api/system-monitor/diag/execute
GET  /api/system-monitor/diag/audit
```

平台 MQTT 消费模块会接收系统监测、告警、诊断和配置拉取回复，并写入本地记录，供页面查询和审计。

检查服务：

```bash

journalctl -u system-monitor@monitor-service.service -n 100 --no-pager
```

检查 MQTT 配置：

```bash
grep -n "systemMonitorRequestTopic\\|configPullRequestTopic\\|diagRequestTopic" \
  /opt/modbus-gateway/config/runtime/apps/monitor-service.json
```

检查直连维护监听：

```bash
ss -lntp | grep 9443
```

检查实时租约：

```bash
cat /opt/modbus-gateway/run/realtime-meter-leases.json
```

检查配置拉取分片大小：

```bash
grep -n "configPullChunkBytes" /opt/modbus-gateway/config/runtime/apps/monitor-service.json
```

## 13. 当前结论

当前推荐生产模式：

- 远端维护、配置拉取、诊断和平台监测走 MQTT。
- Win 工具在本地网口直连时走 `SystemMonitor` 内嵌维护 API。
- 普通业务全量上传仍由 `MqttDriver` 按 `fullUploadIntervalMs` 发布。
- 操作员打开实时监控时才启用高频实时点位监听。
- 控制类通道使用 QoS 2，并由共享内存写队列和驱动执行结果回执闭环。
