# 摄像头接入推流设计

## 目标

摄像头接入用于把边端视频画面推送到平台，实现远程预览、录像、截图和状态监测。

设计边界：

- 视频流不进入共享内存。
- 视频流不通过 MQTT 传输。
- MQTT 只承载摄像头状态、事件和控制命令。
- 摄像头在线状态、码率、帧率等轻量指标可以写入共享内存，供平台统一监测。
- 平台侧通过媒体服务播放视频流，不把视频数据存入 MySQL。

## 摄像头来源

边端建议支持三类摄像头：

| 类型 | 来源 | 说明 |
| --- | --- | --- |
| USB UVC | `/dev/video0`、`/dev/video1` | 适合本地 USB 摄像头 |
| 网络摄像头 | `rtsp://ip/stream` + `sourceAuth` | 适合 IPC、NVR 输出 |
| CSI/MIPI | 厂商驱动设备节点 | 后续按实际硬件支持情况接入 |

第一阶段建议优先支持：

- RTSP 网络摄像头拉流。
- USB UVC 摄像头通过 ffmpeg/GStreamer 编码后推流。

## 总体架构

```text
局域网摄像头 / USB 摄像头
        |
        | 边端主动拉流
        v
边端 CameraService
        |
        +--> 主动推流到公网平台媒体入口
        |
        +--> 摄像头状态写入共享内存
        |
        +--> MQTT 上报状态和事件到公网平台

公网平台媒体服务
        |
        v
平台 Web 页面预览 / 录像 / 截图
```

生产拓扑需要明确为：摄像头和边端在同一个现场局域网内，平台在公网环境。平台不能主动访问现场摄像头或边端内网地址，生产链路必须采用边端主动出站连接：

```text
现场摄像头  ->  边端 CameraService  ->  平台固定公网 IP
  内网拉流         本地处理             主动推流
```

公网访问链路：

```text
边端 CameraService
  |
  | 主动推流到平台固定公网 IP
  v
平台公网媒体入口
  |
  +--> 平台媒体服务保存/转码/录像
  |
  +--> 浏览器通过 HTTPS 访问 HLS/WebRTC 播放地址
```

这个模式不要求边端有公网 IP，也不要求现场路由器做端口映射。只要求边端能访问平台固定 IP 的媒体入口端口。

推荐把视频能力拆成独立服务：

```text
camera-service@camera-service.service
```

原因：

- 视频编解码和协议驱动资源模型不同。
- 摄像头异常不应影响 Modbus、DLT645、MQTT 数据链路。
- 后续可以独立限制 CPU、内存和重启策略。
- 视频服务可以独立 OTA 配置。

## 推流方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| RTSP | 延迟低，工业设备常见 | 浏览器不能直接播放 | 边端到平台媒体服务 |
| HLS | 浏览器兼容好 | 延迟较高，通常 2 到 10 秒 | 平台网页预览、低实时性 |
| WebRTC | 浏览器低延迟 | 架构复杂，需要信令和 NAT 处理 | 低延迟预览 |
| MJPEG | 实现简单 | 带宽高，CPU 占用高 | 调试和低帧率预览 |

推荐分阶段：

1. 第一阶段：边端推 RTSP 到平台媒体服务，平台转换成 HLS 或 WebRTC 给浏览器播放。
2. 第二阶段：如需要低延迟，平台引入 WebRTC 服务。
3. 第三阶段：按需增加本地录像、事件截图、云端录像。

## 推荐生产方案

生产环境建议使用：

```text
边端 CameraService + 平台媒体服务
```

当前推荐平台媒体服务选型为 ZLMediaKit，具体部署、端口、Nginx 反代、推流测试和鉴权接入见 [ZLMediaKit平台媒体服务接入教程.md](ZLMediaKit平台媒体服务接入教程.md)。

边端职责：

- 管理摄像头配置。
- 拉取 USB/RTSP 视频源。
- 推流到平台指定地址。
- 监测摄像头在线、帧率、码率。
- 把摄像头状态写入共享内存和 MQTT。

平台职责：

- 保存摄像头配置。
- 下发配置 OTA。
- 提供播放地址。
- 提供录像、截图、权限控制。
- 提供视频预览页面。

不建议：

- 不建议把视频二进制通过 MQTT 发布。
- 不建议 Java 后端直接转发大流量视频。
- 不建议把录像直接放 MySQL。
- 不建议平台直接访问边端内网 RTSP 地址，因为公网和 NAT 环境下不可控。

## 公网访问模式

平台有固定公网 IP 时，建议采用：

```text
摄像头 -> 边端: 现场 LAN RTSP/USB 拉流
边端 -> 平台固定公网 IP:8554 RTSP 推流
平台 -> 浏览器:443 HTTPS/HLS/WebRTC 播放
边端 -> 平台 MQTT Broker:1883/8883 状态和控制
```

边端配置只需要改媒体服务入口：

```json
{
  "cameraService": {
    "enabled": true,
    "media": {
      "type": "rtsp_push",
      "serverUrl": "rtsp://203.0.113.10:8554",
      "transport": "tcp",
      "reconnectIntervalMs": 5000,
      "auth": {
        "enabled": true,
        "mode": "basic",
        "username": "GW0001",
        "password": "change_me",
        "hideCredentialsInStatus": true
      }
    },
    "cameras": [
      {
        "cameraCode": "CAM001",
        "sourceType": "rtsp",
        "source": "rtsp://192.168.1.50:554/stream1",
        "sourceAuth": {
          "enabled": true,
          "mode": "basic",
          "username": "camera_user",
          "password": "camera_password",
          "hideCredentialsInStatus": true
        },
        "stream": {
          "path": "GW0001/CAM001",
          "publishUrl": ""
        }
      }
    ]
  }
}
```

说明：

- `203.0.113.10` 是文档示例 IP，现场替换为平台固定公网 IP 或域名。
- `stream.path` 必须带上 `machineCode/cameraCode`，避免多网关、多摄像头路径冲突。
- `publishUrl` 留空时，边端自动拼接为 `media.serverUrl + "/" + stream.path`。
- `media.auth.mode=basic` 时，边端会自动把认证信息拼成 `rtsp://username:password@host:port/path`。
- `media.auth.hideCredentialsInStatus=true` 时，MQTT 状态上报中的 `streamUrl` 会隐藏账号密码。
- 平台浏览器不要直接打开边端地址，应打开平台媒体服务生成的 HLS/WebRTC 地址。

平台侧需要开放：

| 方向 | 端口 | 用途 |
| --- | --- | --- |
| 边端到平台 | TCP `8554` | RTSP 推流入口，端口可按媒体服务实际配置调整 |
| 边端到平台 | TCP `1883` / `8883` | MQTT 状态、控制、OTA，生产建议 `8883` TLS |
| 浏览器到平台 | TCP `443` | 平台页面、视频播放、鉴权代理 |

如果公网运营商或防火墙对 RTSP 不友好，第二阶段建议支持 SRT 或 RTMPS 推流。当前第一版已实现 RTSP push，适合作为固定 IP 场景的先行方案。

## 认证模型

摄像头链路有两类认证，不能混用：

- `cameras[].sourceAuth`: 边端访问现场局域网摄像头时使用，例如 IPC 的 RTSP 用户名密码。
- `cameraService.media.auth`: 边端向公网平台媒体入口推流时使用，例如平台给该 `machineCode` 分配的推流账号或 token。

`sourceAuth` 只影响输入源 URL，`media.auth` 只影响输出推流 URL。平台浏览器播放时走平台自己的登录态和 HTTPS 播放鉴权，不使用摄像头源密码。

## 摄像头源认证

网络摄像头在现场局域网内时，建议不要把账号密码直接写进 `source`，而是拆到 `sourceAuth`：

```json
{
  "cameraCode": "CAM001",
  "sourceType": "rtsp",
  "source": "rtsp://192.168.1.50:554/stream1",
  "sourceAuth": {
    "enabled": true,
    "mode": "basic",
    "username": "camera_user",
    "password": "camera_password",
    "hideCredentialsInStatus": true
  }
}
```

实现说明：

- `sourceAuth.mode=basic` 时，边端会在执行 ffmpeg 前生成 `rtsp://camera_user:camera_password@192.168.1.50:554/stream1`。
- `sourceAuth.mode=token_query` 时，边端会把 token 追加到输入源 URL query 中。
- `command` 自定义推流命令优先级最高，配置后不会自动改写输入源和输出源，必须由配置方自行写完整认证参数。

## 平台推流入口认证

平台公网媒体入口必须启用认证，至少满足：

- 不允许匿名写流。
- 每个 `machineCode` 使用独立账号或 token。
- 流路径必须包含 `machineCode/cameraCode`。
- 平台侧校验账号只能写入自己 `machineCode` 下的路径。
- 平台侧保留推流连接日志，记录来源 IP、账号、stream path、上线和断开时间。

边端当前支持两种认证模式：

| 模式 | 配置 | 生成效果 | 适用场景 |
| --- | --- | --- | --- |
| `basic` | `username/password` | `rtsp://user:pass@host:port/path` | mediamtx/ZLMediaKit 等常见 RTSP 入口 |
| `token_query` | `token/tokenParam` | `rtsp://host:port/path?token=xxx` | 平台媒体服务支持 query token 校验 |

`basic` 样例：

```json
{
  "cameraService": {
    "media": {
      "serverUrl": "rtsp://203.0.113.10:8554",
      "auth": {
        "enabled": true,
        "mode": "basic",
        "username": "GW0001",
        "password": "change_me",
        "hideCredentialsInStatus": true
      }
    }
  }
}
```

`token_query` 样例：

```json
{
  "cameraService": {
    "media": {
      "serverUrl": "rtsp://203.0.113.10:8554",
      "auth": {
        "enabled": true,
        "mode": "token_query",
        "token": "edge_stream_token",
        "tokenParam": "token",
        "hideCredentialsInStatus": true
      }
    }
  }
}
```

实现约束：

- 如果 `stream.publishUrl` 已经配置完整地址，认证仍会按 `media.auth` 处理。
- 如果完整 `publishUrl` 已经包含 `user:pass@`，`basic` 模式不会重复注入。
- `CameraService` 日志和 MQTT 状态会隐藏 `sourceAuth` 和 `media.auth` 中的认证信息。
- `command` 自定义命令不自动注入 `sourceAuth` / `media.auth`，适合完全自定义 ffmpeg 或 GStreamer 管线。
- 由于 `ffmpeg` 推流 URL 本身需要认证参数，边端本机 `ps` 仍可能看到进程参数。生产上应限制普通用户登录边端，后续可进一步改成更强的凭据注入方式。

## 边端配置样例

当前 app 配置文件：

```text
config/runtime/apps/camera-service.json
```

当前边端已实现 `CameraService` 可执行程序和 `camera-service@.service` 模板。出厂配置中 `cameraService.enabled=false`，不会默认启动。启用条件为：

- `config/runtime/apps/camera-service.json` 存在。
- `cameraService.enabled=true`。
- `cameraService.cameras[]` 中至少存在一个 `enabled=true` 的摄像头。
- 边端已安装 `ffmpeg`。

关闭行为：

- `cameraService.enabled=false` 时，`gateway-services.sh` 不会启动 `camera-service@camera-service.service`。
- 即使手工执行 `CameraService --app-config ...`，程序也会打印 disabled 后直接退出。
- 共享内存路由不会注册摄像头状态点，避免平台误认为摄像头点位仍属于运行态。
- Java 平台 `/api/camera/streams` 返回 `enabled=false` 和空 `streams`，不会查询 ZLMediaKit，也不会生成运行态播放地址。
- 单个 `cameras[].enabled=false` 时，该摄像头不会被边端推流，也不会在平台运行态查询中合并 ZLMediaKit/共享内存状态。

## 平台播放地址

平台应按 `machineCode + cameraCode` 管理摄像头。

平台保存字段：

| 字段 | 说明 |
| --- | --- |
| `machineCode` | 网关唯一标识 |
| `cameraCode` | 摄像头唯一标识 |
| `name` | 摄像头名称 |
| `sourceType` | `usb`、`rtsp`、`csi` |
| `streamPath` | 平台媒体服务中的流路径 |
| `rtspUrl` | 服务端录像或转码使用 |
| `hlsUrl` | 浏览器 HLS 播放地址 |
| `webrtcUrl` | 浏览器 WebRTC 播放地址 |
| `online` | 摄像头在线状态 |
| `lastSeen` | 最近一次状态上报时间 |

示例播放地址：

```text
RTSP:  rtsp://203.0.113.10:8554/GW0001/CAM001
HLS:   https://platform.example.com/video/GW0001/CAM001/index.m3u8
WebRTC:https://platform.example.com/video/GW0001/CAM001/webrtc
```

## MQTT 状态与控制

摄像头 topic 继续按 machineCode 定向，避免所有网关同时收到命令。

状态上报：

```text
edge/camera/status/{machineCode}
```

控制请求：

```text
edge/camera/request/{machineCode}
```

控制回执：

```text
edge/camera/reply/{machineCode}
```

状态样例：

```json
{
  "machineCode": "GW0001",
  "cameraCode": "CAM001",
  "online": true,
  "fps": 15,
  "bitrateKbps": 1480,
  "streamUrl": "rtsp://203.0.113.10:8554/GW0001/CAM001",
  "ts": 1777536000000
}
```

控制样例：

```json
{
  "requestId": "CAM_CMD_1777536000000",
  "machineCode": "GW0001",
  "cameraCode": "CAM001",
  "command": "restart_stream",
  "ts": 1777536000000
}
```

回执样例：

```json
{
  "requestId": "CAM_CMD_1777536000000",
  "machineCode": "GW0001",
  "cameraCode": "CAM001",
  "success": true,
  "message": "stream restarted",
  "ts": 1777536000500
}
```

## 共享内存状态点

视频流不进共享内存，但状态指标可以作为普通点位写入：

| 指标 | 建议 dataType | 说明 |
| --- | --- | --- |
| `camera_online` | `online_status` | 摄像头在线状态 |
| `camera_fps` | `double` | 实际帧率 |
| `camera_bitrate_kbps` | `double` | 实际码率 |
| `camera_error_code` | `int32` | 最近错误码 |

这样平台实时监测、告警、变位和本地画面都可以复用现有点位体系。

## 录像与截图

录像建议由平台媒体服务负责：

- 平台统一配置录像策略。
- 录像文件放对象存储、NAS 或专用磁盘目录。
- MySQL 只保存录像索引、时间范围、文件路径、摄像头编码。

边端本地录像只作为可选能力：

- 适用于断网期间临时缓存。
- 必须配置磁盘配额。
- 使用环形目录，避免长期运行撑爆磁盘。

截图建议由平台媒体服务或边端 CameraService 按需生成：

- 平台主动请求截图。
- 告警联动截图。
- 摄像头离线前后截图。

## 安全设计

- 摄像头账号密码不应明文展示在前端列表。
- 平台下发配置时可以加密敏感字段，边端解密使用。
- 外网访问视频时建议走平台反向代理，不直接暴露边端 RTSP 端口。
- WebRTC/HLS 对外访问需要鉴权 token。
- MQTT 控制 topic 必须按 `machineCode` 定向。
- 生产环境建议启用 MQTT TLS 和平台 HTTPS。
- 平台公网 RTSP 推流入口必须配置账号、路径 ACL 或防火墙白名单，不能裸露匿名写流。
- 边端 `media.auth.password`、`media.auth.token` 属于敏感字段，Java 前端应做密码框和脱敏展示，接口返回时也应避免明文回显。

## 边端服务

当前已安装服务：

```text
camera-service@camera-service.service
```

当前已落地：

- `CameraService --app-config /opt/modbus-gateway/config/runtime/apps/camera-service.json`
- 通过 `ffmpeg` 托管 USB/RTSP/test 视频源。
- 推流进程异常退出后按 `media.reconnectIntervalMs` 自动重启。
- 摄像头在线、fps、bitrate、errorCode 写入 `cameraService.sharedMemoryName`。
- 摄像头状态按 `statusIntervalMs` 发布到 `cameraService.statusTopic/{machineCode}`。
- `gateway-services.sh` 会按配置自动判断是否启动 `camera-service@camera-service.service`。

如使用独立媒体组件，可增加：

```text
mediamtx.service
```

服务管理逻辑：

- `cameraService.enabled=false` 时不启动。
- 没有摄像头配置时不启动。
- 没有任何 `enabled=true` 的摄像头时不启动。
- `cameraService.enabled=false` 时不注册摄像头共享内存状态点。
- 摄像头异常时只重启 CameraService，不影响采集驱动。
- OTA 配置更新后重启 CameraService 和相关媒体服务。

## Java 平台改造点

当前已落地的 Java 平台能力：

- 左侧菜单新增“摄像头监控”。
- `MQTT / OTA / App` 页面新增“摄像头”子页，编辑独立文件 `runtime/apps/camera-service.json`。
- 支持新增、删除、复制摄像头。
- 支持配置 RTSP、USB、UVC、test 源。
- 支持分别配置 `sourceAuth` 和 `cameraService.media.auth`，避免把现场摄像头认证和公网推流认证混用。
- 支持配置分辨率、帧率、码率、编码方式。
- 支持配置共享内存状态点 `online/fps/bitrateKbps/errorCode`。
- 配置 OTA 默认包含 `runtime/apps/camera-service.json`。
- OTA service-manager 包默认包含 `camera-service@.service`。
- 保存 `runtime/device_identity.json` 后，平台会同步 `mqtt-service.json`、`monitor-service.json`、`camera-service.json` 中的 MQTT `clientId` 为当前 `machineCode`。

当前已落地的平台接口：

```http
GET /api/camera/streams?machineCode=GW0001&appName=runtime/apps/camera-service.json
```

该接口从 `camera-service.json` 解析摄像头配置，按 `machineCode/cameraCode` 生成流路径和播放地址，并合并实时点缓存中的摄像头状态点：

- `camera_online` 来自 `statusPointIndexes.online`。
- `camera_fps` 来自 `statusPointIndexes.fps`。
- `camera_bitrate_kbps` 来自 `statusPointIndexes.bitrateKbps`。
- `camera_error_code` 来自 `statusPointIndexes.errorCode`。

如果 `cameraService.enabled=false`，接口只返回服务关闭状态和空 `streams`，前端显示“摄像头服务未启用”。此时平台不会访问 ZLMediaKit，也不会用旧共享内存状态误判摄像头在线。

平台端仍不直接承载 RTSP/HLS/WebRTC 媒体流。实际播放地址由公网媒体服务提供，Java 页面只负责配置、状态展示和跳转播放地址。

后续增强能力：

- 摄像头重启推流、截图、启停等控制命令。
- 录像列表和录像检索。
- HLS/WebRTC 播放鉴权代理。
- 媒体服务推流路径 ACL 和账号生命周期管理。

## 测试流程

USB 摄像头检测：

```sh
ls -l /dev/video*
v4l2-ctl --list-devices
v4l2-ctl -d /dev/video0 --list-formats-ext
```

RTSP 摄像头检测：

```sh
ffprobe rtsp://user:password@192.168.22.50:554/stream1
```

USB 推流测试：

```sh
ffmpeg -f v4l2 -framerate 15 -video_size 1280x720 -i /dev/video0 \
  -c:v libx264 -preset veryfast -tune zerolatency -b:v 1500k \
  -f rtsp rtsp://203.0.113.10:8554/GW0001/CAM001
```

RTSP 转推测试：

```sh
ffmpeg -rtsp_transport tcp -i rtsp://user:password@192.168.22.50:554/stream1 \
  -c copy -f rtsp rtsp://203.0.113.10:8554/GW0001/CAM002
```

平台播放测试：

```text
VLC 打开 rtsp://203.0.113.10:8554/GW0001/CAM001
浏览器打开平台 HTTPS HLS 或 WebRTC 地址
```
