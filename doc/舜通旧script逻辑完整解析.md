## 旧运行模型

旧服务是一个 Qt `QCoreApplication`，入口是 `main.cpp -> Script::Script() -> script.exec()`。

核心数据不来自 JSON 配置，而来自实时数据库共享内存：

- 旧工程 `maxvarindex` 当前为 `4599`，用于创建共享内存数据区。迁移规范把 `4500` 段固定给 LED，结算/并网表改到 `4600` 段，因此新工程若启用结算/并网表需把点位容量扩展到至少 `4699`。
- `QRamRT::GetItemValue/SetItemValue` 读写普通实时量和中间变量。
- 控制命令通过 `executeCmd()` 写入 `KcData` 并置 `KcFlag=1`，外部驱动消费后再清标志。
- `LEDControl()` 也是通过同一命令通道写 RGB 三元组。
- `GetSysItem(2/3/4/5)` 分别被用作日、时、分、秒。

迁移时要保留“普通点写入”和“设备控制命令”两个语义。PCS 启停、PCS 分相 P/Q 下发、风机、断路器等不能简单当成普通 `SetItemValue`。

### BMS 重复寄存器说明

点表中 `8197/8198/8199` 这类寄存器大量重复不是简单重复点。它们是同一告警字的不同 bit：

- `8197`：轻度告警字。
- `8198`：中度告警字。
- `8199`：重度告警字。
- 同一个寄存器配不同 bit 位，例如单体欠压、过压、电流、温差等。

迁移成当前网关点位时，应解析为“同一寄存器 + bit 位”的多个数字量，不能按寄存器地址去重丢点。

## 主运行流程

### 初始化

`Script::Script()`：

1. 读取应用目录下 `maxvarindex`。
2. 创建共享内存。
3. 初始化命令通道 `KcFlag=0`。
4. 创建并连接 `FastTimer/AvgTimer/COS_Timer/LV_Timer/HV_Timer/DS_Timer/PH_Timer/GF_Timer`。
5. 执行 `setUp()`。
6. `FastTimer->start(2000)`，进入 2 秒快循环。

`setUp()`：

- 延迟 10 秒等待采集数据。
- 清空台区、储能表滑动平均队列。
- 若今日充/放电量存档为 0，则用累计充/放电量初始化 `398/399`。

### Fast_Run 快循环

`Fast_Run()` 是主调度入口，每 2 秒执行一次。

执行顺序：

1. 根据整柜自动和 PCS 并网状态计算 `Flag_Auto_All`。
2. `Reflesh_Realtime_Variables()` 读取参数、刷新 LED、回写上一周期策略输出到 `601-632`。
3. `Fault_Alarm()` 汇总 PCS/BMS/液冷/UPS 告警故障，并转换部分云平台状态点。
4. `CLOUD_Change()` 清平台故障分类，处理强制满充反馈，按上一周期 P/Q 输出启停 PCS 并更新 `100` 总动作反馈。
5. 启动 `AvgTimer`，每 2 秒计算滑动平均。
6. IO 自动逻辑：风机启停、充/放/静置状态标志。
7. 急停判断。当前 `ESTOP=0`，即 `992 == 0` 触发急停，触发后转手动、置 `921`、分闸并直接返回。
8. 通讯掉线检查：台区表、储能表、PCS、BMS、制冷、消防。
9. 自动模式：
   - 首次自动时把 PCS/BMS/IO/空调/除湿置自动。
   - 先下发 PCS P/Q 全 0，确认控制量清零后 `Flag_Auto_only=1`。
   - 启动液冷远程控制。
   - 读取电压阈值、台区限制等参数。
   - 调用 `POWER_SOLVE()` 合成并下发上一轮策略输出。
   - 调度各策略定时器或独占模式。
10. 手动模式：
    - 停止各策略定时器。
    - 清 EN/RUN 反馈和所有 `OUT_*`。
    - 下发 PCS P/Q 全 0。
    - 控制量清零后停 PCS。

### Avg_Run 平均值计算

`Avg_Run()` 每 2 秒执行。平均窗口来自 `H_EMS_avg_num=156`。

当前配置下：

- 台区表读取 `1030-1051`，计算并回写 `201-225` 中实际用到的电压、P/Q、COS 和不平衡度。
- 储能表读取 `1130-1143`，计算并回写 `251-266`。
- 因 `Meter_FH=0` 且 `Meter_BW=0`，负荷按 `FH_avg = TQ_avg - CN_avg` 计算，并回写 `309-325`。
- BMS_MODEL 为 2 时，今日充放电量直接依赖 BMS 点；BMS_MODEL 为 1 或 3 时脚本用累计电量差值计算今日电量。

`H_EMS_avg_num` 必须有有效正值。若为 0，旧代码会对空队列执行 `pop_front()`，存在运行风险。

## 策略逻辑

旧逻辑内部有功符号约定：

- `P > 0`：充电。
- `P < 0`：放电。
- `Q > 0`：释放无功，等效电容方向。

### POWER_SOLVE 统一合成

`POWER_SOLVE()` 是多数策略的最终合成和限幅入口。

优先级和处理顺序：

1. 若 `BMS_Charge2Full=true`，直接三相按 `PCS_P1_MAX * 2 / 3` 充电并返回。
2. 无功先取 `OUT_Q*_COS`。
3. 有功基准取 `OUT_P*_DS`。
4. 低压策略输出为放电时，取更负值。
5. 高压策略输出为充电时，取更大正值。
6. 光伏策略输出为充电时，取更大正值。
7. 受控模式运行时，P/Q 三相均分覆盖当前输出。
8. 三相平衡输出叠加到当前 P。
9. 正向/反向台区功率限制修正。
10. 动态增容修正。
11. PCS 分相有功、分相无功、分相视在容量限幅。
12. BMS 允许充/放功率限幅。
13. SOC 上下限保护：低 SOC 禁放，高 SOC 禁充，回差 1%。
14. 通过 `PCS_Power_download()` 下发 PCS P/Q。

`POWER_SOLVE()` 的调试和最终输出点由 `Reflesh_Realtime_Variables()` 写到 `601-632`，但由于刷新在快循环前半段执行，这些点通常反映上一周期计算结果。

### COS 无功补偿

启动条件：`Flag_Auto_only=1`、`H_CL_COS_EN=1`、`BMS_SOC >= BMS_SOCmin`。

参数：

- `501` 扫描时间，工程和代码实际按秒。
- `514` 目标功率因数。
- `504` 单相无功最大值，最终在 `POWER_SOLVE()` 限幅。

计算：

- `target_Qx = abs(tan(acos(target_cos)) * TQ_avg_Px)`。
- 台区无功大于目标时输出补偿无功。
- `H_CL_COS_start=513` 被读取过，但当前计算未真正用它作为启动阈值。

输出：`OUT_QA/QB/QC_COS`、`OUT_Q3_COS`，反馈 `8`。

### 低电压补偿

启动条件：`Flag_Auto_only=1`、`H_CL_LV_EN=1`。

逻辑：

- `CN_avg_Ux < LV_Low` 时，`OUT_Px_LV -= Grad_P`，通过放电抬升电压。
- `CN_avg_Ux > LV_Up` 时，`OUT_Px_LV += Grad_P`，向 0 回退。
- 钳位到 `[-PCS_P1_MAX, 0]`。
- `POWER_SOLVE()` 中若瞬时电压低于 `LV_Low - 8`，输出再乘 `1.2`。

输出：`OUT_PA/PB/PC_LV`、`OUT_P3_LV`，反馈 `10`。

### 高电压补偿

启动条件：`Flag_Auto_only=1`、`H_CL_HV_EN=1`。

逻辑：

- `CN_avg_Ux > HV_Up` 时，`OUT_Px_HV += Grad_P`，通过充电降低电压。
- `CN_avg_Ux < HV_Low` 时，`OUT_Px_HV -= Grad_P`，向 0 回退。
- 钳位到 `[0, PCS_P1_MAX]`。
- `POWER_SOLVE()` 中若瞬时电压高于 `HV_Up + 8`，输出再乘 `1.2`。

输出：`OUT_PA/PB/PC_HV`、`OUT_P3_HV`，反馈 `12`。

### 充电模式 CD

启动条件：`Flag_Auto_only=1`、`H_CL_CD_EN=1`。

它是独占直控模式：启动时关闭 COS/LV/HV/FD/DS/PH/GF/ZR，函数内直接下发 PCS，不走 `POWER_SOLVE()` 的有功合成。

逻辑：

- 要求 `BMS_SOC < H_CL_CD_target_SOC` 且目标功率非 0。
- 若正向台区限制使能：`P_YX = TQ_P3XZ_pos_Value - FH_avg_P3`，输出为 `min(P_YX, target_P)`，小于 0 则置 0。
- 否则按目标功率三相均分充电。

输出：`OUT_P3_CD`，直接下发 `P_A/B/C = OUT_P3_CD / 3`，反馈 `14`。

### 放电模式 FD

启动条件：`Flag_Auto_only=1`、`H_CL_FD_EN=1`、`BMS_SOC >= BMS_SOCmin`。

它也是独占直控模式：启动时关闭 COS/LV/HV/CD/DS/PH/GF/ZR，函数内直接下发 PCS。

逻辑：

- 要求 `BMS_SOC > H_CL_FD_target_SOC` 且目标功率非 0。
- 若反向限制使能：`OUT_P3_FD = -min(FH_avg_P3, target_P)`。
- 否则 `OUT_P3_FD = -target_P`。
- 三相均分放电。

输出：`OUT_P3_FD`，直接下发 `P_A/B/C = OUT_P3_FD / 3`，反馈 `16`。

### 定时充放电 DS

启动条件：`Flag_Auto_only=1`、`H_CL_DS_EN=1`。

启动时读取三组 24 小时表：

- `400-423`：每小时功率。
- `424-447`：每小时目标 SOC。
- `760-783`：每小时模式，`0=自动`、`1=只充电`、`2=只放电`。

当前实现是按当前小时取阶梯值，不做曲线插值。

逻辑：

- 若 `BMS_SOC < 当前小时SOC` 且模式不是只放电，按 `Grad_P` 逐步充电。
- 若 `BMS_SOC > 当前小时SOC` 且模式不是只充电，按 `Grad_P` 逐步放电。
- 达到 SOC 或模式禁止时，按 `2 * Grad_P` 回退到 0。
- 受 `463/464` 电压上下限约束。
- 超过小时功率时，三相设为 `±0.3333 * power`。

输出：`OUT_PA/PB/PC_DS`、`OUT_P3_DS`，反馈 `18`。

重启恢复注意：工程里 `615-618` 有周期存盘，但 C++ 进程启动后并没有从这些点恢复 `OUT_PA/PB/PC_DS` 本地变量，因此旧服务重启后 DS 梯度状态实际会从 0 重新爬坡。新实现若要求异常重启后连续执行，应显式持久化并恢复运行态。

### 三相平衡 PH

启动条件：`Flag_Auto_only=1`、`H_CL_PH_EN=1`。

逻辑：

- `TQ_CN_Px = TQ_avg_Px + CN_avg_Px`，表示排除储能影响前的台区负荷。
- 允许不平衡：`Allow = H_CL_BPH_Per * TQ_avg_P3 / 300`。
- 若 `max - min > Allow`，计算 `SetVal = (max - min - Allow) / 2`。
- 最大相输出 `+SetVal`，最小相输出 `-SetVal`，中间相为 0。

输出：`OUT_PA/PB/PC_PH`、`OUT_P3_PH`，反馈 `20`。

### 光伏 GF

启动条件：`Flag_Auto_only=1`、`H_CL_GF_EN=1`、`Meter_TQ=1`。

逻辑：

- 仅在 `581 <= 当前小时 <= 583` 的充电时间窗动作。
- 若 `FH_avg_Px <= TQ_P1XZ_neg_Value`，则 `OUT_Px_GF = TQ_P1XZ_neg_Value - FH_avg_Px`，用于吸收反送或低负荷。
- 否则输出 0。

源码注释提到“夜晚放电”，但当前 `GF_Run()` 没有实现夜晚放电逻辑，`584/585/586` 在源码中未使用。

输出：`OUT_PA/PB/PC_GF`、`OUT_P3_GF`，反馈 `22`。

### 动态增容 ZR

动态增容没有独立定时函数，而在 `POWER_SOLVE()` 内处理。

条件：`H_CL_ZR_EN=1` 且 `Meter_TQ=1`。

逻辑：

- `588` 是每相保留有功容量。
- 当 `TQ_P1XZ_neg_Value - FH_avg_Px - reserve < 0` 时，说明该相超出限制，需要 PCS 放电补偿。
- 对应相输出取更负值，并置 `24` 动态增容反馈。

### 受控模式 SK

受控版点表已补充 `25/26/590/591`。

启动条件：`Flag_Auto_only=1`、`H_CL_SK_EN=1`、SOC 在 `[BMS_SOCmin, BMS_SOCmax]`。

逻辑：

- 启动时关闭其他策略 EN。
- `SK_Mode()` 读取 `590` 总有功和 `591` 总无功到 `OUT_P3_SK/OUT_Q3_SK`。
- `POWER_SOLVE()` 中只有 `H_CL_SK_RUN == 1` 时，才把 P/Q 三相均分覆盖当前输出。

迁移时需要注意旧源码没有在 `SK_Mode()` 内显式把 `26` 置为 1。如果平台只写 `25` 而不写 `26`，`POWER_SOLVE()` 的覆盖条件可能不生效。建议新逻辑改为 EN 触发后由服务自己维护 RUN。

### 强制满充

源码定义了：

- `H_CL_Charge2Full_RUN = 26`
- `H_CL_Charge2Full_Date1 = 165`
- `H_CL_Charge2Full_Date2 = 166`

但基础工程和受控版点表都没有 `165/166`，且 `26` 已被受控模式反馈占用。

当前有效触发只有“待机模式每月指定日期充满”：

- 自动就绪。
- COS/LV/HV/DS/PH/GF 都未使能。
- 当前日等于 `165` 或 `166`。
- `BMS_SOC < 100`。
- 当天未触发过。

执行时 `POWER_SOLVE()` 直接按 `PCS_P1_MAX * 2 / 3` 三相充电，SOC 到 100 后复位。

源码中“单体压差告警强制充满”和“单体欠压告警强制充满”两段已被注释，不生效。

迁移建议：强制满充不要继续复用 `26`，应独立成 `charge_to_full.enabled/running/date1/date2`，并校验点表存在。

## 故障、告警和平台转换

`Fault_Alarm()` 做以下转换：

- PCS：
  - 当前模型 3 按 `1341/1342` 判断总故障/告警。
  - 云运行状态写到 `1216`，枚举 `1=停机`、`2=运行`、`3=待机`。
  - 云并网状态写到 `1217`，枚举 `1=离网`、`2=并网`。
- BMS：
  - `1491/1492/1493` 任一为 1 则 BMS 告警。
  - `1462` 为 1 则 BMS 故障。
  - `1454` 转换出 `1455/1456` 簇充/放状态。
  - `1550/1554/1454/1462` 转换出 `1551` 簇状态。
- 液冷：
  - ZL=21 时 `4013 != 0` 视为液冷告警，故障固定 false。
- UPS：
  - 当前 `UPS_MODEL=0`，UPS 分支不编译。
- 平台故障分类：
  - `903-920` 每周期被清 0。
  - 急停写 `921`。

已发现风险：`BOX_alarm = BMS_alarm && PCS_alarm && UPS_alarm && XF_alarm && ZL_alarm`，`BOX_fault` 同理。这意味着必须所有设备同时告警/故障才算总告警/故障；当前 UPS/XF 默认为 false 时总故障很难置位。若业务含义是“任一设备故障即柜级故障”，迁移时应改为 OR。

## IO、液冷和 LED

### IO 自动

当 `H_BOX_IO_Auto=1`：

- 读取 PCS 温度 `1270`。
- 温度大于等于 `163` 且小于 200 时，风机 `991` 置 1。
- 温度小于等于 `164` 且大于 -200 时，风机 `991` 置 0。
- 读取 BMS 充/放状态，设置充电、放电、静置标志。

急停触发时：

- `6` 整柜自动置 0。
- `921` 系统急停置 1。
- `flag_run=0`、`flag_fault=1`。
- 若断路器反馈未合适，控制 `985` 分闸/复位。

### 液冷

自动模式下，按不同 `ZL_MODEL` 控制远程开关和运行状态。当前 `ZL_MODEL=21` 时，若 `4000` 为 0 则写 1 启动远程控制。

### LED 屏

当前 `LED_SCREEN=1`：

- `4500` 显示 SOC 数字，实际发送格式为 `"SOC*10 1"`。
- 根据运行、停止、待机、故障、充电、放电、静置等标志，对 `4525/4528/4534/4531/4543/4540/4594/4588/4591` 等 RGB 地址写颜色。

迁移后 `4500-4599` 只归 LED 屏使用，结算/并网表点位统一放到 `4600-4699`。

## 迁移必须保留的行为

1. 设备角色语义：EMS 本体、PCS、BMS、台区表、储能表、液冷/温控、除湿、消防、UPS、DIO。
2. 命令写语义：PCS/IO/液冷控制必须走命令通道或当前网关等价的设备写命令。
3. 功率符号：内部 P 正充电、负放电。
4. 分相控制：PCS 控制是 A/B/C 三相 P/Q，不是单总功率点。
5. 统一仲裁：多数策略只产生 `OUT_*`，最终由统一功率求解做台区、PCS、BMS、SOC 限幅。
6. 独占策略：CD/FD 是直接下发的独占模式，迁移时应明确是否保留这种绕过合成的行为，或统一改成仲裁模式。
7. 滑动平均：台区/储能电表平均和负荷派生是后续策略输入，不能省略。
8. BMS 允许功率：优旦 BMS 用 `实时电压 * 允许电流 * 0.001` 计算允许充/放功率，并回写 `1552/1553`。
9. SOC 禁充禁放：带 1% 回差，且禁放时会清低压和无功运行反馈，禁充时会清高压和光伏运行反馈。
10. 24 小时 DS 表：功率、SOC、模式三组都要保留；新 UI 可做曲线，但提交时要能落成等价的小时段或更细粒度计划。
11. 重启恢复：旧代码没有可靠恢复本地策略变量，新实现应显式持久化运行态。

## 建议的新 EMS 配置模型

为了契合当前 gateway 设计，建议不继续“套壳”旧 `var+index`，而是迁移成以下结构：

### 设备角色

EMS 模式下先配置设备类型，再绑定点位：

- `ems`：EMS 本体，中间变量、策略状态、平台故障。
- `pcs`：PCS，必选启动、停机、分相 P/Q 控制、运行、并网、告警、故障、通讯。
- `bms`：BMS，必选 SOC、实时电压、允许充/放电流或功率、告警、故障、通讯。
- `meter.grid`：台区表，必选三相 P/Q、电压、功率因数、通讯。
- `meter.ess`：储能表，必选三相 P/Q、电压、通讯。
- `meter.load`：负荷表，可选；没有时由台区表和储能表派生。
- `meter.settlement`：结算表，可选，默认使用 `4600-4699` 段，不允许和 LED `4500-4599` 段冲突。
- `thermal`：液冷/空调。
- `dehumidifier`：除湿，可选，未配置时跳过除湿联动和告警。
- `fire`：消防。
- `ups`：UPS，可选，未配置时跳过 UPS 通讯、告警和故障聚合。
- `dio`：急停、断路器、风机、门禁等。

### 语义点

每种设备维护固定语义点，例如：

- PCS：`start_cmd`、`stop_cmd`、`p_a_set`、`p_b_set`、`p_c_set`、`q_a_set`、`q_b_set`、`q_c_set`、`run_status`、`grid_status`、`fault_total`、`alarm_total`、`comm_status`。
- BMS：`soc`、`voltage`、`charge_current_limit`、`discharge_current_limit`、`charge_power_limit`、`discharge_power_limit`、`charge_state`、`discharge_state`、`fault_total`、`alarm_total`、`comm_status`。
- 台区/储能/负荷表：`ua/ub/uc`、`pa/pb/pc/p3`、`qa/qb/qc/q3`、`cosa/cosb/cosc/cos3`、`comm_status`。
- EMS 策略：`enable`、`running`、`output`、`last_run_at`、`parameters`。

平台提交配置时按语义点校验，不再要求脚本里硬编码的地址名完全一致。

### 运行态持久化

新服务需要单独持久化：

- 当前启用策略。
- DS 当前计划、当前小时/段、当前三相输出、上次执行时间。
- LV/HV/COS/PH/GF/ZR/SK 的 `OUT_*`。
- `BMS_low_limit`、`BMS_up_limit`。
- `BMS_Charge2Full`、`Flag_one_day`。
- 滑动平均窗口，或至少重启后的预热规则。
- 最后一次 PCS 下发值和命令确认状态。

这样设备异常重启后才能回滚到重启前状态并继续执行，而不是从 0 输出重新爬坡。

## 后续实现建议

1. 先建立 EMS 设备角色和语义点 profile，覆盖当前舜通工程的 PCS/BMS/电表/液冷/消防/DIO。
2. 导入舜通点表时生成语义点绑定，提交前提示缺失点、冲突点、类型/读写方向错误。
3. 把旧策略拆成原生策略模块：平均值、故障转换、PCS 命令、功率仲裁、DS、COS、LV/HV、PH、GF、ZR、SK、强制满充。
4. DS UI 可以做 0-24 小时折线曲线，但边端存储应保存明确的分段计划和运行态。
5. 对 `CD/FD` 决定是继续独占直控，还是迁移为统一仲裁输入。若要减少复杂度，建议也改成仲裁输入。
6. 对旧工程地址段冲突按 `4500-4599=LED`、`4600-4699=结算/并网表` 迁移，对 `26` 双语义、`BOX_fault` 聚合方式做显式规则，避免旧问题带入新系统。
