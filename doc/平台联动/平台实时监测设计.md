# 平台实时监测设计

本文说明当前 Java 平台实时监测链路。平台实时推送能力已经落地，当前实现位于平台项目 `org.example.realtime` 包，并通过 MQTT 消费、内存缓存、短趋势缓冲和 WebSocket/REST 接口对前端提供实时点位。

## 1. 当前目标

平台实时监测用于查看边端当前采集值，而不是直接操作边端共享内存。

当前覆盖：

- 按 `machineCode`、`meterCode`、`index` 查看实时点位。
- 按 `machineCode / meterCode` 分组展示。
- 接收分片快照并刷新实时缓存。
- WebSocket 推送点值变化。
- REST 查询当前点值和短趋势。
- 按页面请求触发边端实时快照。
- 可选择是否接收全量上传数据进入实时缓存。

不在当前实时监测链路内完成：

- 长期历史曲线存储。
- 业务报表。
- 任意边端 shell 诊断。
- 摄像头视频流承载。

## 2. 当前代码位置

平台模块：

```text
src/main/java/org/example/realtime
```

关键类：

| 类 | 职责 |
| --- | --- |
| `RealtimeMqttConsumer` | 订阅边端 MQTT telemetry、告警、状态 topic |
| `RealtimeMessageIngestService` | 解析 payload、分片快照、点值消息并写缓存 |
| `RealtimePointCacheService` | 保存最新点值，支持按网关/设备/index 查询 |
| `RealtimeTrendBufferService` | 保存短趋势环形缓冲 |
| `RealtimeWebSocketConfig` | 注册 `/ws/realtime` |
| `RealtimeWebSocketHandler` | 接收页面订阅条件 |
| `RealtimePushService` | 按订阅条件向 WebSocket 会话推送 |
| `RealtimeController` | 提供 `/api/realtime/*` REST 接口 |

## 3. Topic 分流

当前实时和全量已经分 topic。

| 类型 | 默认基础 topic | 实际 topic 示例 |
| --- | --- | --- |
| 实时请求 | `edge/telemetry/realtime/request` | `edge/telemetry/realtime/request/COMM202600102` |
| 实时上行 | `edge/telemetry/realtime` | `edge/telemetry/realtime/COMM202600102` |
| 全量上行 | `edge/telemetry/full` | `edge/telemetry/full/COMM202600102` |
| 告警 | `edge/alarm` | `edge/alarm/COMM202600102` |
| 变位 | `edge/event/change` | `edge/event/change/COMM202600102` |
| 状态 | `edge/status` | `edge/status/COMM202600102` |
| 系统监测点位 | `edge/system/monitor/points` | `edge/system/monitor/points/COMM202600102` |

平台默认订阅实时 topic、告警、变位和状态。全量上传可能是一秒级大流量，默认不进入实时缓存；只有配置：

```json
{
  "realtime": {
    "ingestFullUpload": true
  }
}
```

时，平台才订阅 `edge/telemetry/full/+`。

## 4. 边端数据来源

边端有两条实时相关链路：

```text
操作员打开实时监控
  -> 平台 / Win 工具发送订阅或实时请求
  -> SystemMonitor 写 realtime meter lease
  -> MqttDriver 或 SystemMonitor 按表计范围读取共享内存
  -> edge/telemetry/realtime 或 edge/system/monitor/points
  -> 平台实时缓存
```

```text
周期全量上传
  -> MqttDriver 按 fullUploadIntervalMs 扫描共享内存
  -> edge/telemetry/full
  -> 平台按 ingestFullUpload 决定是否进入实时缓存
```

设计要求：

- `realtime` 只在打开实时监控或存在租约时高频发送。
- `full` 按边端配置的 `fullUploadIntervalMs` 定时发送。
- 点位名称不要求边端 MQTT 每次携带；平台和 Win 工具应优先用已拉取工程配置补齐名称。
- 数据 payload 中如包含 `name`，平台会接收；如不包含，页面用工程/配置侧点表补展示名。

## 5. Payload 支持

平台支持两类输入：

### 5.1 快照格式

```json
{
  "type": "snapshot",
  "machineCode": "COMM202600102",
  "chunked": true,
  "chunkId": "1780000000000",
  "chunkIndex": 1,
  "chunkCount": 3,
  "meters": [
    {
      "meterCode": "PCS_001",
      "values": [
        {
          "index": 110001,
          "pointCode": "active_power",
          "value": 12.3,
          "quality": 1,
          "ts": 1780000000000
        }
      ]
    }
  ]
}
```

### 5.2 单点格式

```json
{
  "type": "point_realtime",
  "machineCode": "COMM202600102",
  "meterCode": "PCS_001",
  "index": 110001,
  "pointCode": "active_power",
  "value": 12.3,
  "quality": 1,
  "ts": 1780000000000
}
```

解析兼容字段：

- `index`、`pointIndex`、`idx`、`address`
- `meterCode`、`device`、`deviceId`、`devCode`、`slaveCode`
- `pointCode`、`code`、`point`、`tag`、`metric`
- `name`、`pointName`、`displayName`

## 6. 分片处理

边端大快照会按 `maxPayloadBytes` 分片。平台当前处理策略：

- 收到任意分片，先解析该分片内点值并刷新最新值缓存。
- WebSocket 可立即推送该分片点值，不等待所有分片到齐。
- 同时按 `machineCode + chunkId` 维护分片组装状态。
- 当分片到齐后，按本次快照包含的 `meterCode/index` 清理旧点，避免旧点残留。
- 如果不是分片快照，则直接按本次完整快照清理旧点。

这样可以兼顾低延迟和配置变更后的缓存清理。

## 7. 缓存模型

最新值缓存 key：

```text
machineCode:meterCode:index
```

缓存内容：

- `machineCode`
- `meterCode`
- `index`
- `pointCode`
- `name`
- `value`
- `quality`
- `ts`
- `chunked/chunkId/chunkIndex/chunkCount`
- 收到时间

短趋势缓冲按同一个 key 存储最近 N 个点，当前默认由 `realtime.trendBufferSize` 控制。

缓存会过滤明显旧的数据，避免乱序分片或重复消息覆盖较新的点值。

## 8. WebSocket 接口

WebSocket 地址：

```text
/ws/realtime
```

页面连接后发送订阅条件：

```json
{
  "machineCode": "COMM202600102",
  "meterCode": "PCS_001",
  "index": 110001
}
```

字段规则：

- `machineCode` 必填。
- `meterCode` 可选。
- `index` 可选。
- 字段不允许包含 MQTT 通配符 `+/#`。
- 单条订阅消息最大 `4096` 字节。

推送格式：

```json
{
  "type": "point_batch",
  "points": [
    {
      "type": "point_realtime",
      "machineCode": "COMM202600102",
      "meterCode": "PCS_001",
      "index": 110001,
      "pointCode": "active_power",
      "value": 12.3,
      "quality": 1,
      "ts": 1780000000000
    }
  ]
}
```

## 9. REST 接口

当前平台接口：

```http
GET  /api/realtime/points?machineCode=COMM202600102
GET  /api/realtime/points?machineCode=COMM202600102&meterCode=PCS_001
GET  /api/realtime/points?machineCode=COMM202600102&meterCode=PCS_001&indexes=110001,110002
GET  /api/realtime/trend?machineCode=COMM202600102&meterCode=PCS_001&index=110001
POST /api/realtime/subscribe
POST /api/realtime/unsubscribe
```

`/api/realtime/subscribe` 会调用系统监测命令链路向边端请求实时快照。`/api/realtime/unsubscribe` 当前返回请求范围提示，实时快照主要按请求和租约控制。

## 10. 页面交互规则

页面推荐流程：

1. 选择或输入 `machineCode`。
2. 优先拉取工程配置，构建 `meterCode/index -> name/unit/category` 映射。
3. 打开实时监控，选择要监听的表计。
4. 平台发送实时订阅或快照请求。
5. 页面先用配置点表渲染空值行。
6. MQTT 实际数据到达后刷新对应行。
7. 未收到数据的点继续显示为空值或离线状态，不因为没数据就从表格消失。

该规则同样适用于 Win 端工具：工程配置负责“有哪些点”，实时数据负责“这些点当前值是多少”。

## 11. 配置项

平台读取 app 配置中的 `realtime` 节点：

```json
{
  "realtime": {
    "enabled": true,
    "ingestFullUpload": false,
    "realtimeTelemetryTopic": "edge/telemetry/realtime",
    "fullTelemetryTopic": "edge/telemetry/full",
    "alarmTopic": "edge/alarm",
    "statusTopic": "edge/status",
    "maxLatestPoints": 100000,
    "trendBufferSize": 300,
    "pushThrottleMs": 200
  }
}
```

MQTT 基础连接仍来自 `mqtt` 节点，TLS、账号、broker 和 clientId 由平台 MQTT helper 统一处理。

## 12. 安全与容量边界

- 前端不直接连接 MQTT Broker。
- WebSocket 建连走平台鉴权拦截器。
- 页面订阅只能看到平台允许的 `machineCode`。
- MQTT payload 超过 `RealtimeMessageIngestService.MAX_PAYLOAD_BYTES` 会被拒绝。
- 单次 REST `indexes` 过滤最多 `2000` 个点。
- `full` topic 默认不进入实时缓存，避免一秒全量上传时拖慢平台。

## 13. 和系统监测的关系

`SystemMonitor` 负责：

- 接收平台实时订阅请求。
- 写实时表计租约。
- 发布 `edge/system/monitor/points/{machineCode}` 点位快照。
- 提供配置拉取和诊断能力。

`RealtimeMqttConsumer` 负责：

- 消费 MQTT 实时/全量/告警/状态消息。
- 写平台内存缓存。
- 推送 WebSocket。

两者通过 MQTT 解耦，Java 不直接读边端共享内存。

## 14. 运维排查

检查平台是否启用实时 MQTT：

```bash
grep -n "realtime" config/runtime/apps/mqtt-service.json
```

检查当前缓存：

```http
GET /api/realtime/points?machineCode=COMM202600102
```

检查 WebSocket：

```text
ws://<platform-host>/ws/realtime
```

检查边端 topic：

```bash
mosquitto_sub -h <broker> -t 'edge/telemetry/realtime/COMM202600102' -v
mosquitto_sub -h <broker> -t 'edge/system/monitor/points/COMM202600102' -v
```

如果页面无数据，优先确认：

1. 边端是否有共享内存值。
2. 是否已打开实时监控或存在实时租约。
3. 边端是否发布到 `realtime` 或 `system/monitor/points` topic。
4. 平台 app 配置中 `realtime.enabled` 是否为 `true`。
5. 平台是否开启了 TLS 并能连接 broker。
6. `machineCode` 是否和 topic 后缀一致。

## 15. 当前结论

当前正式方案不是前端轮询，而是：

```text
边端共享内存
  -> MqttDriver/SystemMonitor
  -> MQTT realtime/full/monitor points topic
  -> Java RealtimeMqttConsumer
  -> RealtimePointCacheService + RealtimeTrendBufferService
  -> REST / WebSocket
  -> 前端实时监测页面
```

后续增强方向：

- 长期历史曲线接入时序库。
- 多实例平台使用 Redis 共享实时缓存。
- 按用户权限细化 WebSocket 订阅过滤。
- 实时监测和配置工程进一步绑定，减少页面重复拉取配置。
