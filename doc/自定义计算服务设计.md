# 自定义计算服务设计

## 1. 目标

新增独立边端服务 `ComputeEngine`，从一个或多个共享内存读取点位，按配置表达式计算结果，再按输出点路由写回对应共享内存中的虚拟计算点。

目标：

- 计算逻辑与 Modbus / DLT645 / 后续协议驱动解耦。
- 计算服务不依赖 MQTT 扫描周期，直接通过 `PointStoreRouter` 读写共享内存。
- 支持多串口、多协议、多共享内存场景。
- 计算结果作为普通点位进入共享内存，可继续参与实时上传、全量上传、变位上传、告警判断、本地存储和平台监测。
- 第一阶段支持安全表达式 DSL；复杂 native plugin 放到后续阶段。

## 2. 关键修正：必须支持多共享内存

多串口场景下，一个网关可能存在多个共享内存，例如：

```text
gateway_point_store_ttysp1
gateway_point_store_ttysp2
gateway_point_store_ttysp3
gateway_point_store_compute
```

因此 `ComputeEngine` 不能假设所有输入和输出都在 `gateway_point_store`。正确模型是：

```text
协议驱动 A -> shm A
协议驱动 B -> shm B
协议驱动 C -> shm C

ComputeEngine
  -> PointStoreRouter
  -> 按 index 读取 shm A/B/C
  -> 按输出 index 写回目标 shm
```

设计规则：

- 输入点通过 `PointStoreRouter::routeByIndex()` 定位所在共享内存。
- 输出点优先通过 `PointStoreRouter::routeByIndex()` 定位目标共享内存。
- `latestOnly` 输出点找不到路由时，可按 `outputs[].sharedMemoryName` 或 `outputDefaultSharedMemoryName` 自动注册为 `meterCode=COMPUTE` 的虚拟路由。
- `deviceWrite` 输出点必须已经存在于物理设备点表中，并且 `write.enable=true`，否则拒绝下发。
- 计算点推荐定义在 `device_compute_*.json` 中并加入 app 配置的 `deviceConfigFiles[]`，这样平台、MQTT、告警、变位和存储能获得完整点位元数据。
- 如果计算输出分属多个共享内存，应拆成多个计算设备配置文件。

## 3. 总体架构

```text
ModbusRtu / Dlt645Driver / 后续协议驱动
        |
        | putLatest 原始采集点
        v
多个共享内存
        ^
        | route by index
        |
PointStoreRouter
        ^
        | read inputs / write outputs
        |
ComputeEngine
        |
        | expression / plugin
        v
虚拟计算点 device_compute_*.json
        |
        v
EventEngine / MqttDriver / SystemMonitor
```

`ComputeEngine` 是独立进程，不嵌入 MQTT，不嵌入协议驱动。`MqttDriver`、`EventEngine`、`SystemMonitor` 通过同一份 `deviceConfigFiles[]` 识别物理点和计算点。

## 4. 服务边界

新增进程：

```text
ComputeEngine
```

启动方式：

```bash
/opt/modbus-gateway/bin/ComputeEngine \
  --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json
```

systemd 模板：

```text
compute-engine@.service
```

`gateway-services.sh` 启动规则：

- `computeEngine.enabled=true` 时启动 `compute-engine@mqtt-service.service`。
- 不存在 `computeEngine` 或 `enabled=false` 时不启动。
- 协议驱动不启动 `protocol.type=computed` 的虚拟设备文件。

## 5. App 配置设计

建议放在 `runtime/apps/mqtt-service.json`：

```json
{
  "identityConfigFile": "/opt/modbus-gateway/config/runtime/device_identity.json",
  "deviceConfigFiles": [
    "/opt/modbus-gateway/config/runtime/devices/device_slave_ttySP1.json",
    "/opt/modbus-gateway/config/runtime/devices/device_slave_ttySP2.json",
    "/opt/modbus-gateway/config/runtime/devices/device_compute_global.json"
  ],
  "computeEngine": {
    "enabled": true,
    "sharedMemoryNames": [
      "gateway_point_store_ttysp1",
      "gateway_point_store_ttysp2",
      "gateway_point_store_compute"
    ],
    "outputDefaultSharedMemoryName": "gateway_point_store_compute",
    "scanIntervalMs": 200,
    "maxRuleEvalPerScan": 1000,
    "badQuality": 0,
    "defaultOutputTtlMs": 600000,
    "rules": []
  }
}
```

字段说明：

| 字段 | 含义 |
| --- | --- |
| `enabled` | 是否启用计算服务 |
| `sharedMemoryNames` | 计算服务打开的共享内存列表；为空时从 `mqttDriver.sharedMemoryNames[]` 和 `deviceConfigFiles[]` 的 `memoryStore.sharedMemoryName` 自动合并 |
| `outputDefaultSharedMemoryName` | `latestOnly` 输出点找不到既有路由时使用的默认共享内存 |
| `scanIntervalMs` | 计算扫描周期 |
| `maxRuleEvalPerScan` | 单轮最多执行规则数，防止长时间占用 CPU |
| `badQuality` | 输入无效或计算失败时输出质量码，建议 `0` |
| `defaultOutputTtlMs` | 计算输出默认 TTL |
| `rules` | 计算规则列表 |

## 6. 计算点配置

计算点放在虚拟设备配置文件中：

```text
runtime/devices/device_compute_global.json
runtime/devices/device_compute_ttySP1.json
runtime/devices/device_compute_ttySP2.json
```

示例：

```json
{
  "schemaVersion": "1.0.0",
  "protocol": {
    "type": "computed"
  },
  "collect": {
    "defaultIntervalMs": 1000,
    "batchOptimize": false,
    "maxBatchRegisters": 1
  },
  "memoryStore": {
    "sharedMemoryName": "gateway_point_store_compute",
    "maxLatestPoints": 100000,
    "maxPendingWrites": 4096,
    "maxPersistentSamples": 20000
  },
  "meters": [
    {
      "meterCode": "CALC",
      "deviceName": "计算点",
      "points": [
        {
          "index": 900001,
          "pointCode": "pump_efficiency",
          "name": "泵效率",
          "category": "computed",
          "enabled": true,
          "isStore": true,
          "fullUpload": true,
          "reportOnChange": true,
          "persistIntervalSec": 60,
          "read": {
            "enable": true,
            "function": 0,
            "length": 0,
            "dataType": "computed",
            "unit": "%",
            "intervalMs": 1000,
            "cachePolicy": {
              "ttlMs": 600000
            }
          },
          "write": {
            "enable": false
          },
          "alarms": []
        }
      ]
    }
  ]
}
```

约束：

- `protocol.type=computed` 表示该文件只定义虚拟计算点。
- 每个 `device_compute_*.json` 只能对应一个 `memoryStore.sharedMemoryName`。
- 计算点 `index` 必须全局唯一，建议使用 `900000` 以上号段。
- 输出要写入哪个共享内存，由计算点所在文件的 `memoryStore.sharedMemoryName` 决定。
- 如果要把计算点放入多个共享内存，就生成多个 `device_compute_*.json`。

推荐输出布局：

| 布局 | 说明 | 适用场景 |
| --- | --- | --- |
| 独立计算共享内存 | 所有计算点写入 `gateway_point_store_compute` | 跨接口汇总、能耗统计、KPI 指标 |
| 就近写入接口共享内存 | `device_compute_ttySP1.json` 写入 `gateway_point_store_ttysp1` | 单接口内派生点、减少共享内存数量 |

## 7. 计算规则配置

示例：

```json
{
  "ruleCode": "pump_efficiency_calc",
  "name": "泵效率计算",
  "enabled": true,
  "trigger": {
    "type": "interval",
    "intervalMs": 1000
  },
  "inputs": [
    {
      "name": "flow",
      "index": 31101,
      "required": true
    },
    {
      "name": "power",
      "index": 32102,
      "required": true
    }
  ],
  "outputs": [
    {
      "name": "efficiency",
      "index": 900001,
      "mode": "latestOnly",
      "sharedMemoryName": "gateway_point_store_compute",
      "ttlMs": 600000,
      "qualityPolicy": "bad_if_any_input_bad"
    },
    {
      "name": "setpoint",
      "index": 31110,
      "mode": "deviceWrite",
      "minIntervalMs": 1000,
      "deadband": 0.1,
      "qualityPolicy": "bad_if_any_input_bad"
    }
  ],
  "script": {
    "type": "expression",
    "expression": "if(valid(flow) && valid(power) && power > 0, flow / power * 100, null)"
  }
}
```

字段说明：

| 字段 | 含义 |
| --- | --- |
| `ruleCode` | 规则唯一编码 |
| `trigger.type` | 触发方式，第一阶段支持 `interval` 和 `onInputChanged` |
| `inputs[].name` | 表达式变量名 |
| `inputs[].index` | 输入点 index，可来自任意共享内存 |
| `inputs[].required` | 输入缺失时是否判定规则失败 |
| `outputs[].index` | 输出 index；`latestOnly` 可由计算服务自动注册虚拟路由，推荐仍写入 `device_compute_*.json`；`deviceWrite` 必须是可写物理点 |
| `outputs[].mode` | 输出模式：`latestOnly`、`deviceWrite`、`both` |
| `outputs[].sharedMemoryName` | `latestOnly` 自动注册路由时的目标共享内存；已有路由时以 Router 路由为准 |
| `outputs[].ttlMs` | 输出值过期时间 |
| `outputs[].minIntervalMs` | 设备写入最小间隔，防止计算结果频繁写设备 |
| `outputs[].deadband` | 设备写入死区，变化小于该值不下发 |
| `qualityPolicy` | 输出质量策略 |
| `script.type` | 第一阶段使用 `expression` |
| `script.expression` | 表达式内容 |

## 8. 表达式能力

第一阶段建议实现受限表达式 DSL，不直接执行任意 Python / JS / Shell。

支持类型：

- 数字：`double`
- 布尔：`true / false`
- 空值：`null`

支持运算符：

```text
+ - * / %
> >= < <= == !=
&& || !
()
```

支持函数：

| 函数 | 含义 |
| --- | --- |
| `if(cond, a, b)` | 条件选择 |
| `valid(x)` | 输入存在、质量正常、未过期 |
| `stale(x)` | 输入是否过期 |
| `quality(x)` | 输入质量码 |
| `abs(x)` | 绝对值 |
| `min(a,b)` | 最小值 |
| `max(a,b)` | 最大值 |
| `round(x,n)` | 保留 n 位小数 |
| `clamp(x,min,max)` | 限幅 |
| `delta(x)` | 后续阶段支持：当前值与上次值差 |
| `rate(x)` | 后续阶段支持：单位时间变化率 |

## 8.1 EMS 领域脚本

除通用 `expression` 外，`ComputeEngine` 还支持 EMS 领域脚本：

| `script.type` | 用途 |
| --- | --- |
| `legacyEms` | 复刻旧舜通 `script.cpp` 逻辑，作为兼容执行器和回归基准 |
| `graphEms` | 加载平台生成的 EMS 策略图 JSON，由边端解释执行 |

`graphEms` 的 app 配置示例：

```json
{
  "ruleCode": "shuntong_graph_ems",
  "enabled": true,
  "trigger": {
    "type": "interval",
    "intervalMs": 1000
  },
  "script": {
    "type": "graphEms",
    "graphFile": "runtime/logic/shuntong_ems_graph.json",
    "graphProfile": {
      "PCS_MODEL": "3",
      "Meter_TQ": "1",
      "Meter_CN": "1"
    }
  }
}
```

`graphEms` 复用本章的共享内存和写设备边界：节点输出 latest 时走 `PointStoreRouter::putLatestByIndex()`；需要实际写 PCS 时，必须由 `pcsWriteback` 提交 `PendingWriteCommand`，再由协议驱动消费。模板默认 `pcsWriteback.submitWrites=false`，用于影子运行和输出对比。

## 9. 触发方式

### interval

按固定周期执行规则。

```json
{
  "trigger": {
    "type": "interval",
    "intervalMs": 1000
  }
}
```

### onInputChanged

输入点值发生变化时执行。

```json
{
  "trigger": {
    "type": "onInputChanged",
    "minIntervalMs": 200,
    "deadband": 0.001
  }
}
```

第一阶段不直接消费 `MemoryPointStore::drainPointUpdates()`，因为 `EventEngine` 已经使用点位更新队列，多个服务同时 drain 会互相抢数据。`ComputeEngine` 自己按 `scanIntervalMs` 读取输入点，并在本进程内保存上次值和时间戳来判断变化。

## 10. 输出动作模型

计算结果有三种输出模式：

| 模式 | 行为 | 适用场景 |
| --- | --- | --- |
| `latestOnly` | 只写共享内存最新值，不下发设备 | 派生 KPI、统计值、平台展示点 |
| `deviceWrite` | 不写计算点，只提交设备写命令 | 自动控制、联动写设定值 |
| `both` | 先写计算点，再提交设备写命令 | 需要平台看到计算结果，同时下发设备 |

结论：

- 写入共享内存 latest 本身不会实际下发到设备。
- 要实际写设备，必须使用 `deviceWrite` 或 `both`。
- 设备写入必须走现有 `PendingWriteCommand` 队列，由目标协议驱动消费后执行。
- `ComputeEngine` 不能直接操作串口、TCP 或 DLT645 客户端。

## 11. 输出写回共享内存

计算服务不直接调用某一个默认 store 的 `putLatest()`。实现时需要给 `PointStoreRouter` 增加写 latest 的接口，例如：

```cpp
PointStoreRouter::putLatestByIndex(PointValue value)
```

执行步骤：

1. 根据 `value.index` 查 `PointStoreRoute`。
2. 找到目标 `sharedMemoryName` 对应的 `MemoryPointStore`。
3. 用路由中的 `machineCode / meterCode / pointCode` 补全 `PointValue`。
4. 调用目标 store 的 `putLatest()`。
5. 找不到路由时，如果是 `latestOnly` 输出，则按输出配置自动注册虚拟路由；如果是设备写入，则拒绝并记录规则错误。

输出点字段来源：

| 字段 | 来源 |
| --- | --- |
| `index` | `outputs[].index` |
| `machineCode` | `device_identity.json` |
| `meterCode` | 输出点路由中的 meterCode |
| `pointCode` | 输出点路由中的 pointCode |
| `value` | 表达式结果 |
| `quality` | 质量策略结果 |
| `ts` | 计算完成时间 |
| `expireAt` | `ts + ttlMs` |
| `stale` | 写入时 false |
| `isStore` | 输出点配置 |
| `persistIntervalSec` | 输出点配置 |

写入后影响：

- `MqttDriver` 下次全量上传会带上计算点。
- 计算点 `reportOnChange=true` 时，`EventEngine` 会生成变位事件。
- 计算点配置 `alarms` 时，`EventEngine` 会做告警判断。
- 计算点 `isStore=true` 时，共享内存持久化队列会产生采样。

## 12. 输出下发设备

当输出 `mode=deviceWrite` 或 `mode=both` 时，计算服务需要提交写设备命令：

```cpp
PendingWriteCommand command;
command.cmdId = "CMP_" + ruleCode + "_" + nowMs;
command.index = output.index;
command.value = computedValue;
command.source = "compute-engine";
command.ts = nowMs;
PointStoreRouter::submitWriteCommand(command);
```

执行链路：

```text
ComputeEngine
  -> PointStoreRouter::submitWriteCommand()
  -> 目标 sharedMemoryName 的 pendingWrites
  -> 对应协议驱动 WritebackService
  -> CommandExecutor
  -> 实际设备
  -> 写成功后协议驱动 putLatest 更新共享内存
```

约束：

- `output.index` 必须是物理设备点位，并且点位配置 `write.enable=true`。
- `PointStoreRoute.writable=false` 时拒绝提交写命令。
- DLT645 当前不支持写入，目标点位为 DLT645 时默认拒绝。
- `deviceWrite` 不直接刷新 latest，避免平台误以为设备已经写成功。
- `both` 可以同时写一个计算展示点和一个设备写点，但建议用两个 output 明确区分。
- 写设备必须配置 `minIntervalMs`，避免每次扫描都重复下发。
- 建议配置 `deadband`，只有计算结果变化超过死区才下发。
- 写失败由协议驱动日志和 MQTT command reply / status 扩展反馈；第一阶段至少记录边端日志。

设备写入保护：

| 保护项 | 说明 |
| --- | --- |
| `minIntervalMs` | 同一规则同一输出点最小写入间隔 |
| `deadband` | 与上次已提交值差值小于死区时不写 |
| `write.enable` | 目标物理点必须显式允许写 |
| `write.minValue/maxValue` | 复用现有写入范围校验 |
| `write.allowedValues` | 复用现有枚举校验 |
| `maxWritesPerScan` | 限制单轮最多提交写设备命令 |

示例：计算温控设定值并下发设备：

```json
{
  "ruleCode": "temperature_setpoint_auto",
  "enabled": true,
  "trigger": {
    "type": "interval",
    "intervalMs": 1000
  },
  "inputs": [
    {
      "name": "roomTemp",
      "index": 31101,
      "required": true
    }
  ],
  "outputs": [
    {
      "name": "setpointWrite",
      "index": 31110,
      "mode": "deviceWrite",
      "minIntervalMs": 5000,
      "deadband": 0.5,
      "qualityPolicy": "bad_if_any_input_bad"
    }
  ],
  "script": {
    "type": "expression",
    "expression": "if(valid(roomTemp), clamp(25 - (roomTemp - 22) * 0.2, 18, 28), null)"
  }
}
```

## 13. 质量策略

建议第一阶段支持：

| 策略 | 含义 |
| --- | --- |
| `bad_if_any_input_bad` | 任一输入质量不正常或 stale，则输出 bad |
| `bad_if_required_missing` | required 输入缺失才输出 bad |
| `always_good` | 只要表达式成功就输出 good |

质量码建议：

| quality | 含义 |
| --- | --- |
| `1` | 正常 |
| `0` | 无效、缺失、stale、表达式异常 |
| `2` | 计算降级值，可选 |

## 14. Java 平台改造

平台端新增“计算服务”配置能力：

- 启用/禁用 `computeEngine.enabled`。
- 从现有点表中选择输入点，输入点可跨串口、跨共享内存。
- 新增计算点时选择输出共享内存，默认使用 `outputDefaultSharedMemoryName`。
- 自动生成或维护 `device_compute_*.json`。
- 输出 index 从专用号段自动分配，并检查全局唯一。
- 表达式保存前做变量、函数、输出点和共享内存路由校验。
- 生成配置 OTA 时带上 `runtime/apps/mqtt-service.json` 和所有 `device_compute_*.json`。

页面校验重点：

- 输入 index 必须能在已加载设备配置中找到。
- 输出 index 必须存在于某个计算点文件中。
- 输出点所属共享内存必须出现在 `computeEngine.sharedMemoryNames[]` 或可由 app 自动扫描得到。
- 禁止输出点与物理采集点 index 冲突。
- 第一阶段禁止规则环，例如 A 输出作为 A 输入。

## 16. 测试方案

### 单规则跨共享内存测试

1. `ttySP1` 驱动写入输入点 `31101` 到 `gateway_point_store_ttysp1`。
2. `ttySP2` 驱动写入输入点 `32102` 到 `gateway_point_store_ttysp2`。
3. `ComputeEngine` 读取两个输入点。
4. 输出点 `900001` 写入 `gateway_point_store_compute`。
5. 用 `pointctl --app-config ... snapshot` 确认输入和输出都能查到。

### 设备下发测试

1. 选择一个 Modbus 可写点，例如 `31110`，确保 `write.enable=true`。
2. 配置计算规则输出 `mode=deviceWrite`，目标 index 为 `31110`。
3. 启动 `ComputeEngine` 和对应 `ModbusRtu` 驱动。
4. 观察 `ModbusRtu` 日志是否消费 pending write 并实际发写寄存器报文。
5. 用设备侧或读回点确认写入结果。

### 变位和告警联动测试

1. 输出计算点配置 `reportOnChange=true` 和告警规则。
2. 改变输入点。
3. 订阅 `edge/event/change/<machineCode>` 和 `edge/alarm/<machineCode>`。
4. 确认输出值变化时只触发一次变位，越限时触发告警。

### 性能测试

建议压测维度：

- 规则数：100 / 1000 / 5000。
- 每条规则输入数：2 / 4 / 8。
- 共享内存数量：1 / 4 / 8。
- 扫描周期：100ms / 200ms / 1000ms。
- 输出点是否启用 `reportOnChange`。
- 输出点是否启用告警。
- 输出是否包含 `deviceWrite`。

关键指标：

- 单轮计算耗时。
- 跨共享内存读取耗时。
- 输出写回 latest 耗时。
- pending write 提交耗时。
- 实际设备写回成功率。
- CPU 使用率。
- 输出点 stale 数量。
- EventEngine 事件积压情况。

## 17. 风险与限制

| 风险 | 处理 |
| --- | --- |
| `latestOnly` 输出点 index 找不到路由 | 自动注册为 `meterCode=COMPUTE` 的虚拟路由；生产建议仍生成 `device_compute_*.json` 保留完整元数据 |
| `deviceWrite` 输出点 index 找不到路由 | 拒绝写设备，记录日志 |
| 多共享内存配置漏填 | 启动时从 app 的 `deviceConfigFiles[]` 自动补齐 |
| 输出点和物理点 index 冲突 | Java 保存和边端启动双重校验 |
| 多服务抢 pointUpdates | 第一阶段不 drain pointUpdates，只轮询 latest |
| 表达式异常 | 输出 bad quality 并记录日志，不退出进程 |
| 规则过多导致 CPU 高 | `maxRuleEvalPerScan`、扫描周期、压测准入 |
| 计算结果频繁写设备 | `minIntervalMs`、`deadband`、`maxWritesPerScan` 限制 |
| 写设备失败但计算值已展示 | `deviceWrite` 不直接刷新 latest；`both` 场景要区分展示点和设备写点 |
| native plugin 崩溃 | 第一阶段不启用；后续需要隔离策略 |

## 18. 实施阶段

第一阶段：

- 新增 `ComputeEngine` 可执行程序。
- 增加 `computeEngine` 配置解析。
- `PointStoreRouter` 增加 `putLatestByIndex()`。
- 复用 `PointStoreRouter::submitWriteCommand()` 支持 `deviceWrite` 输出。
- 支持多共享内存输入和输出路由；`latestOnly` 可自动注册输出虚拟路由。
- 支持 `expression` 表达式。
- 支持 `interval` 和轮询式 `onInputChanged`。
- 支持 `latestOnly / deviceWrite / both` 三种输出模式。
- gateway-services 支持自动启动。
- Java 页面支持计算规则、计算点和设备写入输出配置。

第二阶段：

- 支持多输出表达式。
- 支持跨规则依赖排序。
- 支持设备写入回执和平台展示。
- 支持 native plugin，但默认关闭。
- 增加 plugin sha256 校验和加载目录限制。

第三阶段：

- 计算规则运行统计上报。
- 单规则耗时和错误次数监控。
- 平台端表达式在线试算。
- 配置 OTA 前做全局冲突检查。
- 压测报告补充计算服务容量边界。

## 19. 推荐结论

采用“独立 `ComputeEngine` + `PointStoreRouter` 多共享内存路由 + 表达式 DSL + `device_compute_*.json` 虚拟计算点 + pending write 设备下发”的方案。

写入共享内存 latest 不会自动下发到设备。需要实际写设备时，规则输出必须配置为 `deviceWrite` 或 `both`，由 `ComputeEngine` 提交 `PendingWriteCommand`，再由目标协议驱动完成实际写入。这样既能保证多共享内存路由正确，也不会绕过现有写入校验和协议驱动安全边界。
