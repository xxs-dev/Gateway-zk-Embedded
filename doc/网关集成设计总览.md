## 3. 总体架构

```text
ModbusRtu / Dlt645Driver / 后续接口驱动
        |
        | write latest values / pending writes / leases
        v
接口级共享内存分片
        |
        | PointStoreRouter 统一路由
        v
统一点位读写视图
        |
        | scan latest values / submit write commands
        v
MqttDriver / EventEngine / SystemMonitor
        |
        | telemetry / alarm / change / command reply / ota status / monitor / config pull
        v
MQTT Broker
        |
        v
Java 平台 / Web 前端
```

关键设计：

- 生产默认使用单网关统一共享内存 `gateway_point_store`，Modbus、DLT645 以及后续协议驱动都写入同一个共享内存。
- `PointStoreRouter` 维护 `index -> sharedMemoryName / machineCode / meterCode / pointCode` 路由，MQTT、事件引擎、pointctl 统一通过 Router 读写；多共享内存只作为后续接口隔离的兼容扩展。
- 多个协议驱动可以并行运行，但每个驱动只消费自己配置内的写入命令。
- MQTT 独立为一个驱动进程，从 `mqttDriver.sharedMemoryNames[]` 聚合读取并统一上传。
- Java 端不直接依赖边端 device 配置解析实时数据，实时 payload 自带 `machineCode / meterCode / pointCode`。
- OTA 配置包可以清理旧运行 JSON，再写入新 JSON，并由 `gateway-services.service` 统一重启所需驱动。
- OTA 默认保留边端已有 `device_identity.json`，只有 manifest 显式声明 `updateIdentity=true` 时才覆盖网关身份。

### 4.2 systemd 服务

边端部署目录：

```text
/opt/modbus-gateway
```

服务模板：

- `modbus-rtu@.service`
- `dlt645-driver@.service`
- `mqtt-driver@.service`
- `event-engine@.service`
- `system-monitor@.service`
- `gateway-services.service`

推荐只启用：

```bash
systemctl enable gateway-services.service
systemctl start gateway-services.service
```

`gateway-services.sh` 会扫描：

```text
/opt/modbus-gateway/config/runtime/apps/mqtt-service.json
/opt/modbus-gateway/config/runtime/apps/monitor-service.json
/opt/modbus-gateway/config/runtime/devices/<app 中 deviceConfigFiles[] 引用的设备文件>.json
```

启动规则：

- app 的 `deviceConfigFiles[]` 引用设备文件且 `protocol.type=modbus_rtu` 或 `modbus_tcp`：启动 `modbus-rtu@<配置文件名>.service`
- app 的 `deviceConfigFiles[]` 引用设备文件且 `protocol.type=dlt645_2007`：启动 `dlt645-driver@<配置文件名>.service`
- `runtime/apps/mqtt-service.json` 中 `eventEngine.enabled=true`：启动 `event-engine@mqtt-service.service`
- `runtime/apps/mqtt-service.json` 中 `mqtt.enabled=true` 且数据上传、事件 outbox、OTA、实时请求或控制请求任一通道启用：启动 `mqtt-driver@mqtt-service.service`
- `runtime/apps/monitor-service.json` 中 `systemMonitor.enabled=true`：启动 `system-monitor@monitor-service.service`

如果某类配置不存在或没有被 app 引用，该类驱动不会启动。

## 6. 运行配置 JSON 设计

### 6.1 网关身份配置

`machineCode` 是网关本机身份，不属于任何串口、协议或从站配置。当前统一存放在：

```text
config/runtime/device_identity.json
```

示例：

```json
{
  "schemaVersion": "1.0.0",
  "machineCode": "GW0001",
  "imei": "860123456789012",
  "serialNumber": "SZR-GW-0001",
  "model": "EDGE-GW",
  "hardwareVersion": "1.0",
  "firmwareVersion": "1.0.0"
}
```

字段说明：

- `machineCode`：平台侧网关唯一编号，用于 MQTT topic、OTA、监测、配置拉取和实时数据分组。
- `imei`：设备蜂窝模块 IMEI，用于资产识别；模块更换后可能变化，因此不建议直接替代 `machineCode`。
- `serialNumber/model/hardwareVersion/firmwareVersion`：设备资产和版本信息。

应用配置通过 `identityConfigFile` 指向该文件。边端进程启动后把 `machineCode` 注入运行时 `DeviceConfig`，设备 JSON 不再维护 `machineCode`。

### 6.2 设备配置顶层结构

设备配置通用结构：

```json
{
  "schemaVersion": "1.0.0",
  "protocol": {},
  "collect": {},
  "memoryStore": {},
  "meters": []
}
```

字段说明：

- `machineCode`：不再放在设备配置中，统一来自 `device_identity.json`。
- `protocol`：协议和传输配置。
- `collect`：采集周期和批量优化策略。
- `memoryStore`：共享内存、持久化、写回队列参数。
- `meters`：从站、仪表和点位。

## 7. 统一共享内存与 Router

当前生产默认所有协议写入同一个 `gateway_point_store`。实际规则是：

- 每个设备配置仍通过 `memoryStore.sharedMemoryName` 指定目标 POSIX 共享内存，但默认值统一为 `gateway_point_store`。
- Java 多串口批量生成默认保持同一共享内存，只通过不同设备配置文件和不同 systemd 实例区分串口/协议。
- `mqtt-service.json` 的 `mqttDriver.sharedMemoryNames[]` 默认只包含 `gateway_point_store`。
- `PointStoreRouter` 对上提供统一取数和写入方法，业务不应直接依赖某一个 shm 名称。
- 兼容字段 `mqttDriver.sharedMemoryName` 仍保留；如未来确实需要按接口隔离，再同时维护 `sharedMemoryNames[]`。
- 驱动注册点位时会检查同一共享内存中是否已有其他活跃进程占用同一个 `index`，发现重复会拒绝启动，避免重复采集写坏实时值。

容量：

- latest slots：`100000`
- pending write slots：`4096`
- persistent sample slots：`20000`
- claim slots：`100000`

运行时检查：

```sh
/opt/modbus-gateway/bin/pointctl stats --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json
```

该命令会输出 latest、pendingWrites、persistent、pointUpdates 的当前占用、配置上限和物理容量。

主要内容：

- latest values：最新点值。
- pending writes：MQTT / pointctl 写入命令。
- persistent samples：待持久化采样。
- leases：点位归属和活跃状态。

写回保护：

- 每个协议驱动只消费自身配置内的写命令。
- Router 按 `index` 找到目标共享内存并提交写命令，避免多个协议驱动抢走不属于自己的写命令。
- 跨共享内存调试建议使用 `pointctl --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json ...`。

```json
    "keepAliveSec": 60,
    "sessionExpirySec": 0
  }

```

支持：

- `mqtt3`
- `mqtt5`
- TCP 直连
- TLS
- 用户名密码
- QoS0 / QoS1
- 实时数据按 `maxPayloadBytes` 自动分片，默认 `49152` 字节
- 请求类 topic 在运行时自动追加 `/<machineCode>`，例如 `edge/ota/request/GW0001`
- 上行类 topic 也会按 `/<machineCode>` 发布，平台侧使用 `edge/telemetry/+` 等通配订阅

TLS 说明：

- 配置 `mqtt.tls.enabled=true` 或 broker 使用 `ssl://` / `tls://` / `mqtts://` 时启用内置 TLS。
- 内置 TLS 运行时动态加载 OpenSSL，边端需具备 `libssl` / `libcrypto`。
- `certFile` / `keyFile` 支持双向 TLS；`insecureSkipVerify=true` 只允许临时联调。
- 如果目标系统无法提供 OpenSSL，可使用 `mqtt-tls-tunnel@.service` 本机 stunnel 兜底。

验证状态：

- 已在边端使用 `ssl://` broker 和 `tls.caFile` 完成 TLSv1.3 握手、MQTT5 连接、QoS1 发布和订阅验证。
- 已回归普通 `tcp://` MQTT 连接。
- 双向 TLS 的证书字段已实现，需在具备客户端证书校验的真实 broker 上做现场验收。

### 8.2 MQTT Driver

```json
{
  "mqttDriver": {
    "enabled": true,
    "sharedMemoryName": "gateway_point_store",
    "sharedMemoryNames": [
      "gateway_point_store"
    ],
    "scanIntervalMs": 500,
    "fullUploadIntervalMs": 1000,
    "snapshotBacklogThreshold": 2000,
    "snapshotBackoffIntervalMs": 5000,
    "eventReplayMaxBytes": 262144,
    "publishFullOnStart": true,
    "publishAllOnFull": true,
    "fullUploadIndexes": []
  }
}
```

说明：

- `scanIntervalMs`：扫描共享内存和处理 MQTT 下发的周期。
- `fullUploadIntervalMs`：全量实时数据上报周期。
- `publishAllOnFull=true`：全量上报全部可见点。
- `fullUploadIndexes`：可指定只上传部分点。
- `sharedMemoryNames[]`：MQTT 聚合读取的共享内存列表；生产默认只写 `gateway_point_store`。
- `snapshotBacklogThreshold` / `snapshotBackoffIntervalMs`：事件 outbox 积压过多时临时推迟全量 snapshot，优先补发告警/变位事件。
- `eventReplayMaxBytes`：事件补发按字节限流，避免补发大批事件时长时间占用 MQTT 发送链路。
- DLT645 电表采用公共标准点表时，平台和边端会按电表序号自动展开标准点 index；模板点未显式配置 `fullUpload=false` 时默认参与全量上传。

## 9. MQTT 消息结构

### 9.1 实时数据

topic：

```text
edge/telemetry/<machineCode>
```

新格式：

```json
{
  "type": "snapshot",
  "machineCode": "GW0001",
  "meters": [
    {
      "meterCode": "TTYSP1_SLAVE0001",
      "values": [
        {
          "index": 31101,
          "pointCode": "reg_1",
          "value": 123,
          "quality": 1,
          "ts": 1776900000000,
          "stale": false
        }
      ]
    }
  ]
}
```

实时数据分片格式：

当一次全量实时数据超过 `mqtt.maxPayloadBytes` 时，边端仍发布到 `edge/telemetry/<machineCode>`，但会拆成多片：

```json
{
  "type": "snapshot",
  "machineCode": "GW0001",
  "chunked": true,
  "chunkId": "1776906000000-1",
  "chunkIndex": 1,
  "chunkCount": 3,
  "meters": [
    {
      "meterCode": "TTYSP1_SLAVE0001",
      "values": [
        {
          "index": 31101,
          "pointCode": "reg_1",
          "value": 123,
          "quality": 1,
          "ts": 1776906000000,
          "expireAt": 1776906600000,
          "stale": false
        }
      ]
    }
  ]
}
```

平台处理规则：

- 实时监测页面可以收到一片就刷新一片，不需要等待所有分片。
- 如果需要完整快照，则按 `machineCode + chunkId` 聚合，收到 `chunkCount` 片后合并。
- `chunkIndex` 从 `1` 开始。
- 分片可能乱序到达，平台不能依赖接收顺序。
- 未分片消息不带 `chunked / chunkId / chunkIndex / chunkCount`。

### 9.2 变位事件

topic：

```text
edge/event/change/<machineCode>
```

触发条件：

- 点位配置 `reportOnChange=true`
- 第一次扫描只建立基线
- 后续只有 `value` 变化才上传
- `ts / expireAt` 刷新不会重复触发

### 9.3 告警事件

topic：

```text
edge/alarm/<machineCode>
```

特点：

- 支持高限、低限、恢复。
- 触发顺序修正为先恢复旧告警，再触发新告警。
- `persistValue` 为空时不存盘；非空时写入告警 SQLite。

### 9.4 命令下发

topic：

```text
edge/command/request/<machineCode>
```

请求：

```json

  "cmdId": "CMD2026041515300001",
  "machineCode": "GW0001",
  "meterCode": "TTYSP1_SLAVE0001",
  "pointCode": "reg_1",

  "value": 200,
  "source": "mqtt",

```

```text

```

说明：

- MQTT 回执表示命令已进入共享写队列或被拒绝。
- 实际设备写入由对应协议驱动消费 pending write 后执行。
- 点位必须 `write.enable=true`。

### 9.5 OTA

请求 topic：

```text
edge/ota/request/<machineCode>
```

状态 topic：

```text
edge/ota/status/<machineCode>
```

回执 topic：

```text
edge/ota/reply/<machineCode>
```

OTA 状态：

- `accepted`
- `downloading`
- `verifying`
- `applying`
- `rollback`
- `completed`
- `failed`

状态 payload 可带进度字段：

```json
{
  "jobId": "OTA_1776841659459",
  "machineCode": "GW0001",
  "stage": "downloading",
  "status": "running",
  "progress": 10,
  "downloadedBytes": 0,
  "totalBytes": 10532,
  "message": "downloading artifact",
  "ts": 1776841660603
}
```

平台会保存并展示 `progress / downloadedBytes / totalBytes`。边端内置 HTTP 下载器在下载过程中按已写入文件的真实字节数周期性上报：

- `downloadedBytes`：已经写入本地 OTA 包的字节数。
- `totalBytes`：优先取 HTTP `Content-Length`，没有时取 OTA 请求里的 `size`。
- `progress`：`downloadedBytes / totalBytes * 100`，只在 `downloading` 阶段代表真实传输百分比。

限制：

- `http://` 使用边端内置下载器，按 socket 接收字节上报。
- `https://` 使用边端 `curl` 或 `wget` 下载，父进程轮询目标文件大小上报真实已落盘字节。
- `https://` 精确百分比依赖 OTA 请求中的 `size` 字段；Java 平台发布配置 OTA 时会携带该字段。
- 内置 `http://` 下载器不支持 HTTP `Transfer-Encoding: chunked`。
- Java 平台生成的配置 OTA 链接应保持完整 `http://<平台IP>:<端口>/api/config/ota/config-packages/...` URL。

## 10. MQTT 断链缓存与补传

当前采用两条链路：

### 10.1 实时数据 Ring

配置：

```json
{
  "offlineBuffer": {
    "enabled": true,
    "mode": "ring",
    "realtimeFile": "/opt/modbus-gateway/data/mqtt-spool/realtime_ring.dat",
    "realtimeFileSizeBytes": 1073741824,
    "maxRealtimeMessageBytes": 4194304,
    "replayBatchSize": 20
  }
}
```

说明：

- 只缓存实时数据。
- 文件启动时预分配 1GB。
- 单条实时消息最大 4MB。
- 文件满后覆盖最旧实时数据。
- 恢复连接后按批次补传。

容量估算：

```text
1GB / 3MB ~= 341 条
1GB / 2MB ~= 512 条
```

### 10.2 SQLite Event Outbox

缓存数据：

- 告警事件
- 变位事件
- OTA 状态

配置：

```json
{
  "eventOutbox": {
    "sqlitePath": "/opt/modbus-gateway/data/mqtt_event_outbox.db",
    "sqliteLibraryPath": "",
    "retentionMonths": 12,
    "cleanupIntervalHours": 24,
    "replayBatchSize": 100
  }
}
```

表结构：

```sql
CREATE TABLE IF NOT EXISTS mqtt_event_outbox (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  event_type TEXT NOT NULL,
  topic TEXT NOT NULL,
  payload TEXT NOT NULL,
  event_ts INTEGER NOT NULL,
  event_month TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  sent INTEGER NOT NULL DEFAULT 0,
  sent_at INTEGER,
  retry_count INTEGER NOT NULL DEFAULT 0,
  last_error TEXT
);
```

清理规则：

- 只清理 `sent=1` 的已发送事件。
- 保留 12 个自然月。
- 例如：`2026-01` 的数据在 `2027-02` 清理。
- `sent=0` 的未补发数据不会按时间清理。

补发顺序：

```text
SQLite Event Outbox -> Realtime Ring -> 当前新消息
```

详细设计：

```text
doc/断链缓存补传设计.md
```

## 12. Java 配置平台

主要功能：

- 图形化编辑 MQTT / OTA / 共享内存配置。
- 多串口批量生成 Modbus / DLT645 配置。

配置文件命名规则：

- 单串口配置：`device_slave_<serial>.json`

## 13. DLT645 标准点表

DLT645-2007 国标点表固定，由模板生成：

```text
config/templates/dlt645_2007_standard_points.json
```

当前模板状态：

- `pointCount=276`
- `pointCode` 已按 `config/samples/messages/VarList.xml` 的 `unit` 字段同步
- 设备在线状态点使用独立数据类型 `device_online`

设计原则：

- 生成 DLT645 配置时同时附带标准点位模板。
- `points=[]` 表示从标准点表自动展开，默认全量上传，不需要在电表下手工新增 DI 点。
- 页面支持“展开标准点表用于裁剪”：展开后生成显式点表，可以删除不需要的点；再次点击“重置为标准点表”会清空显式点表并恢复公共模板全量模式。
- 只保留 4 个费率：尖、峰、平、谷。
- 支持上 N 结算点电能值。
- 不支持的点位直接从模板剔除，而不是禁用。
