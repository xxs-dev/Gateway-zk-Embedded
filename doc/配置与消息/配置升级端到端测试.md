# 配置 OTA 端到端测试说明

本文用于验证平台或 Windows 工具生成配置包后，通过 OTA 链路下发到边端并自动重启目标服务的完整流程。

## 测试目标

验证链路：

```text
配置编辑
  -> 生成配置 OTA 包
  -> 发布 OTA 请求
  -> 边端下载配置包
  -> ota-apply.sh 校验 manifest 并落盘
  -> systemctl daemon-reload
  -> 按 manifest 重启白名单内服务
  -> MQTT 上报 reply/status
  -> 必要时 ota-rollback.sh 回滚
```

## 测试范围

当前测试对象是配置 OTA，不是程序二进制 OTA。

重点验证：

- `config/runtime/devices/*.json`
- `config/runtime/apps/*.json`
- `config/templates/*.json`
- `manifest.json`
- `deploy/ota-apply.sh`
- `deploy/ota-rollback.sh`
- systemd 服务模板和 `gateway-services.sh`

## 前置检查

平台或 Windows 工具应能生成配置 OTA 包，并能提供边端可访问的下载地址。

边端目录：

```bash
mkdir -p /opt/modbus-gateway/config/runtime/devices
mkdir -p /opt/modbus-gateway/config/runtime/apps
mkdir -p /opt/modbus-gateway/config/templates
mkdir -p /opt/modbus-gateway/ota/downloads
mkdir -p /opt/modbus-gateway/ota/staging
mkdir -p /opt/modbus-gateway/ota/backup
```

OTA 脚本：

```bash
install -m 0755 deploy/ota-apply.sh /opt/modbus-gateway/bin/ota-apply.sh
install -m 0755 deploy/ota-rollback.sh /opt/modbus-gateway/bin/ota-rollback.sh
```

服务状态：

```bash
systemctl status gateway-services.service
systemctl status mqtt-driver@mqtt-service.service
systemctl status system-monitor@monitor-service.service
```

## 生成配置包

配置包应至少包含：

- `manifest.json`
- 需要下发的设备配置
- 需要下发的 app 配置
- 需要下发的模板文件

如果配置包还更新服务管理文件，应包含：

- `deploy/gateway-services.sh`
- `deploy/ota-apply.sh`
- `deploy/ota-rollback.sh`
- `deploy/*.service`

`manifest.json` 中的服务重启段示例：

```json
{
  "restart": {
    "enabled": true,
    "services": [
      "gateway-services.service"
    ]
  }
}
```

`ota-apply.sh` 只允许重启白名单内服务，避免配置包误操作系统服务。

## 手工下载验证

发布 OTA 前，先在边端验证下载地址：

```bash
wget "<artifactUrl>" -O /tmp/config-ota.tar.gz
tar -tzf /tmp/config-ota.tar.gz | head
```

如果下载失败，先排查：

- 下载地址是否是边端可访问 IP 或域名。
- 平台反代是否允许该路径。
- 防火墙和路由是否放通。
- 下载地址是否已过期。

## 手工 apply 验证

```bash
/opt/modbus-gateway/bin/ota-apply.sh \
  /tmp/config-ota.tar.gz \
  2026.06.26.001 \
  CONFIG_OTA_TEST_001 \
  /opt/modbus-gateway/ota/backup \
  /opt/modbus-gateway/ota/staging
```

检查暂存目录：

```bash
find /opt/modbus-gateway/ota/staging/CONFIG_OTA_TEST_001 -maxdepth 4 -type f
cat /opt/modbus-gateway/ota/staging/upgrade_history.log
cat /opt/modbus-gateway/ota/staging/current_version.txt
```

检查目标目录：

```bash
find /opt/modbus-gateway/config/runtime/devices -maxdepth 1 -type f -name '*.json' -ls
find /opt/modbus-gateway/config/runtime/apps -maxdepth 1 -type f -name '*.json' -ls
find /opt/modbus-gateway/config/templates -maxdepth 1 -type f -name '*.json' -ls
```

## MQTT 发布验证

订阅 OTA 回执和状态：

```bash
mosquitto_sub -h <broker> -p <port> -t 'edge/ota/reply/+' -v
mosquitto_sub -h <broker> -p <port> -t 'edge/ota/status/+' -v
mosquitto_sub -h <broker> -p <port> -t 'edge/status/+' -v
```

预期回执：

```json
{
  "jobId": "CONFIG_OTA_TEST_001",
  "machineCode": "COMM202600102",
  "accepted": true,
  "message": "accepted"
}
```

预期状态至少包含：

- `accepted`
- `downloading`
- `verifying`
- `applying`
- `completed` 或 `failed`

## 服务重启验证

配置落盘后检查：

```bash
systemctl status gateway-services.service --no-pager
systemctl status mqtt-driver@mqtt-service.service --no-pager
systemctl status system-monitor@monitor-service.service --no-pager
journalctl -u gateway-services.service -n 100 --no-pager
```

如果设备配置被 `deviceConfigFiles[]` 引用，`gateway-services.sh` 会按协议启动对应驱动：

- `modbus-rtu@*.service`
- `dlt645-driver@*.service`
- `dio-driver@*.service`
- `can-driver@*.service`
- `iec-driver@*.service`

未被 app 配置引用的设备文件不会启动，避免初始化包里存在但项目没用到的驱动占用串口。

## 回滚验证

```bash
/opt/modbus-gateway/bin/ota-rollback.sh \
  /tmp/config-ota.tar.gz \
  2026.06.26.001 \
  CONFIG_OTA_TEST_001 \
  /opt/modbus-gateway/ota/backup \
  /opt/modbus-gateway/ota/staging
```

检查：

```bash
cat /opt/modbus-gateway/ota/staging/rollback_CONFIG_OTA_TEST_001.txt
cat /opt/modbus-gateway/ota/staging/rollback_CONFIG_OTA_TEST_001_restored.txt
systemctl status gateway-services.service --no-pager
```

## 常见问题

### 生成成功但边端下载失败

检查：

- `artifactUrl` 是否能从边端访问。
- 平台下载接口是否需要鉴权但 OTA 请求没有携带。
- HTTPS 证书是否被边端信任。
- 代理、Nginx、平台服务端口是否正确。

### 下载成功但 apply 失败

检查：

- 包内是否有 `manifest.json`。
- `manifest.files[].path` 是否是允许的相对路径。
- 目标目录是否可写。
- 边端是否有 `python3`、`tar`。
- 包内模板文件是否完整。

### 配置已替换但服务没重启

检查：

- `manifest.restart.enabled` 是否为 `true`。
- `manifest.restart.services` 是否包含正确服务。
- 服务名是否在 `ota-apply.sh` 白名单内。
- `systemctl daemon-reload` 是否执行成功。

### 回滚没有恢复

检查：

- apply 阶段是否成功创建 backup。
- `backupDir` 是否被清理。
- 回滚脚本参数里的 `jobId` 是否和 apply 时一致。

## 推荐测试顺序

1. 生成配置 OTA 包。
2. 边端手工下载。
3. 边端手工执行 `ota-apply.sh`。
4. 验证配置落盘和服务重启。
5. 通过 MQTT 发布 OTA。
6. 验证 OTA reply/status。
7. 执行 `ota-rollback.sh`。
8. 验证配置恢复和服务状态。

## 当前实现说明

- 配置 OTA 包格式为 `tar.gz`。
- `ota-apply.sh` 按 `manifest.json` 落盘文件。
- `ota-apply.sh` 支持服务管理文件更新、自动 `chmod +x`、`systemctl daemon-reload` 和白名单服务重启。
- `ota-rollback.sh` 支持从备份目录恢复 `/opt` 与 `/etc` 下的配置和服务文件。
- 首次从旧版边端升级时，建议先手工更新 `ota-apply.sh` 和 `ota-rollback.sh`，避免旧脚本不认识当前 manifest 和服务管理文件。

后续增强项：

- rollback 按 manifest 精准恢复。
- 配置页面展示 OTA 包内文件清单。
- MinIO 预签名地址一键生成并发布。
