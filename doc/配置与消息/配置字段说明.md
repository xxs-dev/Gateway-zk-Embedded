
# Config 说明

本文档专门说明 `config/` 目录下各类配置文件的用途，以及主要字段的含义。

## 文件分类

当前 `config/` 目录建议按两类区分：

- 运行时依赖配置
  - `config/runtime/devices/*.json`
  - `config/runtime/apps/*.json`
- 报文与日志样例
  - `config/samples/messages/*.json`
  - `config/samples/logs/*.log`

当前已整理为：

- 设备采集配置
  - `config/runtime/devices/device_slave_ttySP1.json`
  - `config/runtime/devices/device_slave_ttySP2.json`
  - `config/runtime/devices/device_alarm_multi_slave_tcp_1_2.json`
  - `config/runtime/devices/device_dlt645_multi_meter_1_2.json`
- MQTT 驱动 / 应用配置
  - `config/runtime/apps/mqtt-service.json`
  - `config/runtime/apps/monitor-service.json`
  - `config/runtime/apps/camera-service.json`
- MQTT 消息样例
  - `config/samples/messages/mqtt_command_request_example.json`
  - `config/samples/messages/mqtt_telemetry_chunk_example.json`
  - `config/samples/messages/mqtt_ota_request_example.json`
  - `config/samples/messages/mqtt_ota_status_example.json`
  - `config/samples/messages/mqtt_status_started_example.json`
  - `config/samples/messages/mqtt_status_full_snapshot_example.json`
  - `config/samples/messages/mqtt_alarm_event_example.json`

## 1. 设备采集配置

设备配置文件负责定义：

- 协议类型
- 通信参数
- 采集周期
- 共享内存参数
- SQLite 参数
- 从站列表
- 点位读写能力
- 点位告警规则

### 顶层字段

#### `schemaVersion`

配置版本号，用于后续配置演进兼容。

#### `machineCode`

网关编码。  
同一边端实例建议唯一。

#### `protocol`

协议定义。

字段：

- `type`
  - `modbus_rtu`、`modbus_tcp`、`dlt645_2007` 或 `local_dio`
- `slave`
  - 默认从站地址
  - 多从站模式下可被 `devices[].slave` 覆盖
- `transport`
  - RTU 串口参数
  - Modbus RTU 的 `readRetryCount` 是单帧读取的立即重发次数；例如配置为 `1` 时，首次读取超时后会再补发 1 次
  - 只有 `readRetryCount` 耗尽后仍失败，采集器才会进入 `collect` 中的任务/设备退避
- `tcp`
  - TCP 连接参数
- `backend`
  - `local_dio` 使用的硬件访问后端，当前为 `sysfs_gpio`
- `gpioBasePath`
  - `local_dio` 的 GPIO sysfs 根路径，默认 `/sys/class/gpio`

`local_dio` 是本机 DI/DO 独立协议，不使用串口或 TCP。只有设备文件被 `mqtt-service.json` 或 `monitor-service.json` 的 `deviceConfigFiles[]` 引用时，`gateway-services.sh` 才会启动 `dio-driver@<设备文件名>.service`。

#### `collect`

采集线程参数。

字段：

- `defaultIntervalMs`
  - 默认采集周期
- `batchOptimize`
  - 是否按连续地址合并读任务
- `maxBatchRegisters`
  - 单批最大寄存器数
- `cycleBackoffEnabled`
  - 是否启用按采集轮次退避
  - `true` 时，失败退避不再按固定毫秒等待，而是跳过后续若干次调度机会
  - `false` 时，继续使用 `slaveFailureBackoffMs`、`taskFailureBackoffMs`、`realtimeTaskFailureBackoffMs` 等旧毫秒退避字段
- `slaveFailureBackoffThreshold`
  - 同一逻辑设备连续失败达到该次数后，触发整设备退避
- `slaveFailureBackoffCycles`
  - 整设备失败后跳过的采集轮次
  - 轮次按该设备被调度的机会计算，会自然受到同链路设备数量、总读帧数、超时、帧间隔和重发次数影响
- `taskFailureBackoffThreshold`
  - 单个读任务连续失败达到该次数后，触发读任务退避
- `taskFailureBackoffCycles`
  - 普通采集模式下，单个读任务失败后跳过的基础轮次
- `taskFailureBackoffMaxCycles`
  - 普通采集模式下，单个读任务连续失败时允许增长到的最大跳过轮次
  - 当前按 `1 -> 2 -> 4` 这类倍增方式增长，不超过该上限
- `realtimeTaskFailureBackoffCycles`
  - 实时监控模式下，单个读任务失败后跳过的基础轮次
- `realtimeTaskFailureBackoffMaxCycles`
  - 实时监控模式下，单个读任务连续失败时允许增长到的最大跳过轮次

#### `memoryStore`

共享内存和本地缓存参数。

字段：

- `enabled`
  - 是否启用内存存储
- `backend`
  - 当前主要是 `memory`
- `keepHistory`
  - 历史条数逻辑参数
- `defaultTtlMs`
  - 默认缓存有效期
- `indexBy`
  - 逻辑索引键
- `sharedMemoryName`
  - 共享内存段名称；生产默认统一为 `gateway_point_store`
- `maxLatestPoints`
  - latest 点位逻辑上限；运行时会在编译期物理容量内按配置值限制
- `maxPendingWrites`
  - 待写命令逻辑上限；队列满时会拒绝新写命令，不再静默覆盖旧命令
- `maxPersistentSamples`
  - 待落盘采样逻辑上限
- `sqlitePath`
  - 点位分钟存盘 SQLite 路径
- `sqliteLibraryPath`
  - SQLite 动态库路径
- `persistFlushIntervalMs`
  - 采样落盘周期
- `writebackIntervalMs`
  - 写回线程消费周期
- `writebackBatchSize`
  - 单次写回最多消费多少条命令

#### `devices`

多从站设备列表。

每个元素表示一个逻辑设备，字段：

- `meterCode`
  - 设备编码
- `deviceName`
  - 设备名称
- `slave`
  - 对应 RTU slave / TCP unit id
- `points`
  - 点位列表

补充说明：

- 设备采集配置里的逻辑设备分组字段统一为 `meters`
- MQTT 上送的 grouped telemetry / snapshot 消息体分组字段也是 `meters`
- 配置结构和消息结构字段已统一，平台侧不需要再区分 `devices` / `meters`

## 2. 点位字段

### 基础字段

- `index`
  - 共享内存主索引，必须全局唯一
- `pointCode`
  - 点位编码
- `name`
  - 点位名称
- `desc`
  - 说明
- `category`
  - 分类
- `address`
  - Modbus 寄存器地址
- `enabled`
  - 是否启用
- `isStore`
  - 是否按分钟本地存盘
- `reportOnChange`
  - 是否启用变位上传；为 `true` 时首次扫描只建立基线，后续仅在数值真正变化时发送单条 `change` 事件到 `mqtt.changeEventTopic`，不因 `ts / expireAt` 刷新重复上传
- `persistIntervalSec`
  - 本地采样存盘周期
- `tags`
  - 系统自动生成标签，不在模板和导入表中手工填写
  - 当前生成规则：设备在线点生成 `status/online`，变位上传点生成 `change`，全量上传点生成 `full_upload`，配置告警规则生成 `alarm`，可写点生成 `writable`
  - 无以上属性时不输出 `tags` 字段，避免普通测点被误分类

### `read`

定义读能力。

字段：

- `enable`
  - 是否可读
- `function`
  - 读功能码，常见 `3/4`
- `length`
  - 寄存器长度
- `dataType`
  - 数据类型，如 `uint16`
  - 本机 DI/DO 使用 `digital_input` 或 `digital_output`
  - 设备在线状态使用 `device_online`
- `scale`
  - 缩放系数
- `offset`
  - 偏移量
- `byteOrder`
  - 字节序
- `signed`
  - 是否按有符号读
- `unit`
  - 单位
- `intervalMs`
  - 该点采集周期
- `gpio`
  - `local_dio` 点位绑定的 Linux GPIO 编号
- `activeHigh`
  - `true` 表示 GPIO 高电平对应逻辑值 `1`
  - `false` 表示 GPIO 低电平对应逻辑值 `1`
- `debounceMs`
  - `digital_input` 防抖毫秒数，建议干接点输入配置 `30`
- `cachePolicy`
  - 缓存策略

本机硬件默认映射：

| 通道 | GPIO | 逻辑 |
| --- | ---: | --- |
| DI1-DI5 | 224、225、226、227、230 | 湿接点，低有效 |
| DI6-DI18 | 134、135、136、137、138、143、128、139、140、341、340、339、338 | 干接点，低有效；现场验证 DI6-DI13 短接时 GPIO 为 `0` |
| DO1-DO8 | 231、238、239、129、130、131、132、133 | 继电器输出，低有效；平台逻辑 `1=闭合/动作` |

舜通旧工程导入时，`modbusTCP_Reg.xml` 中的本机干接点会生成独立 `device_dio.json`。默认映射规则为旧 `DI0-DI7` 对应硬件 `DI6-DI13`，即旧通道号加 `5` 后再映射 GPIO。

### `write`

定义写能力。

字段：

- `enable`
  - 是否可写
- `function`
  - 写功能码，常见 `6/16`
- `length`
  - 写入寄存器长度
- `dataType`
  - 写入数据类型
  - 本机 DO 使用 `digital_output`
- `scale`
  - 写缩放
- `offset`
  - 写偏移
- `byteOrder`
  - 写字节序
- `min`
  - 最小值
- `max`
  - 最大值
- `step`
  - 步长
- `verifyAfterWrite`
  - 是否写后回读校验
- `verifyDelayMs`
  - 写后延时
- `verifyByRead`
  - 是否通过读回校验

注意：

- 只把 `write.enable=true` 打开还不够
- `function / dataType / byteOrder / length` 至少要配完整

### `alarms`

告警规则数组。

每个元素字段：

- `type`
  - `high` 或 `low`
- `threshold`
  - 阈值
- `reportRecovery`
  - 是否上报恢复
- `persistValue`
  - 非空则告警写 SQLite
  - 空字符串则只上报不存盘

## 3. MQTT 驱动配置

MQTT 驱动配置文件负责定义：

- 要读取哪些设备配置文件
- MQTT 连接参数
- MQTT 上传参数
- 告警存盘参数

### `deviceConfigFiles`

设备配置文件路径数组。

作用：

- MQTT 驱动通过它读取点位告警规则
- 不再依赖独立 `alarmRules`

### `mqtt`

MQTT 连接与 topic 参数。

字段：

- `enabled`
  - 是否启用真实 MQTT
- `protocolVersion`
  - `mqtt3` 或 `mqtt5`
- `broker`
  - broker 地址
- `clientId`
  - 配置值保持为当前网关 `machineCode`；边端子服务和平台端消费者会在运行时追加职责后缀，保证 MQTT 连接唯一
- `username`
  - 用户名
- `password`
  - 密码
- `telemetryTopic`
  - 遥测 topic
- `changeEventTopic`
  - 变位事件 topic；配置了 `reportOnChange=true` 的点位在数值变化时按单条事件消息发送到这里
- `alarmTopic`
  - 告警 topic
- `statusTopic`
  - 驱动状态与运行事件 topic
- `commandRequestTopic`
  - 命令请求 topic
- `commandReplyTopic`
  - 命令回执 topic，当前回执的是“已入共享写队列/被拒绝”，不是设备最终执行结果
- `otaRequestTopic`
  - OTA 请求 topic
- `otaReplyTopic`
  - OTA 受理结果 topic
- `otaStatusTopic`
  - OTA 过程状态 topic，当前支持 `accepted/downloading/verifying/applying/completed/failed`
- `qos`
  - QoS
- `maxPayloadBytes`
  - 实时数据 `telemetry / snapshot` 单条 MQTT payload 目标上限，默认 `49152` 字节；超过后自动按 `machineCode + chunkId + chunkIndex + chunkCount` 分片发送到同一个 `telemetryTopic`
- `cleanSession`
  - 是否清会话
- `keepAliveSec`
  - 心跳秒数
- `sessionExpirySec`
  - MQTT5 会话过期秒数
- `tls.enabled`
  - 是否启用内置 MQTT TLS。也可以通过 `ssl://` / `tls://` / `mqtts://` broker scheme 自动启用
- `tls.caFile`
  - TLS CA 文件路径。为空时使用系统默认 CA 路径；嵌入式生产环境建议显式配置
- `tls.certFile`
  - TLS 客户端证书路径，用于 broker 要求客户端证书的双向 TLS
- `tls.keyFile`
  - TLS 客户端私钥路径，必须和 `tls.certFile` 成对配置
- `tls.insecureSkipVerify`
  - 是否跳过证书和主机名校验。仅允许临时联调，生产禁止开启
- `offlineBuffer`
  - MQTT 断链本地缓冲配置。实时数据写入预分配 ring 文件；告警、变位、OTA 状态写入 SQLite outbox；连接恢复后先补发 outbox，再补发实时 ring。
- `offlineBuffer.enabled`
  - 是否启用断链缓冲
- `offlineBuffer.mode`
  - 当前使用 `ring`
- `offlineBuffer.realtimeFile`
  - 实时数据 ring 文件路径
- `offlineBuffer.realtimeFileSizeBytes`
  - 实时数据 ring 文件大小，默认 1GB
- `offlineBuffer.maxRealtimeMessageBytes`
  - 单条实时消息最大大小，默认 4MB
- `offlineBuffer.flushBatchSize`
  - ring 批量写入/补发控制参数
- `offlineBuffer.flushIntervalMs`
  - 刷盘间隔
- `offlineBuffer.replayBatchSize`
  - 实时 ring 恢复连接后单轮补传条数
- `offlineBuffer.eventOutbox.sqlitePath`
  - 关键事件 outbox SQLite 路径
- `offlineBuffer.eventOutbox.retentionMonths`
  - 已发送事件按自然月保留数量，默认 12；例如 2026-01 数据在 2027-02 清理
- `offlineBuffer.eventOutbox.cleanupIntervalHours`
  - outbox 清理检查间隔
- `offlineBuffer.eventOutbox.replayBatchSize`
  - outbox 恢复连接后单轮补传条数

### `mqttDriver`

MQTT 驱动运行参数。

字段：

- `enabled`
  - 是否启用驱动
- `sharedMemoryName`
  - 要读取的共享内存段
- `scanIntervalMs`
  - 扫描周期
- `fullUploadIntervalMs`
  - 全量上报周期
- `snapshotBacklogThreshold`
  - 待发送事件积压阈值；大于该值时可临时延后 full snapshot，优先补发事件
- `snapshotBackoffIntervalMs`
  - 事件积压时的 full snapshot 回退窗口；通常配合 `snapshotBacklogThreshold` 一起使用
- `eventReplayMaxBytes`
  - 单轮事件补发的最大字节预算；补发顺序按 `alarm` 优先于 `change`
- `publishFullOnStart`
  - 启动时是否立即发全量
- `publishAllOnFull`
  - 全量时是否发全部点
- `fullUploadIndexes`
  - 指定全量上传的点索引

### `alarmStore`

告警 SQLite 存储参数。

字段：

- `enabled`
  - 是否启用告警存盘
- `sqlitePath`
  - 告警 SQLite 路径
- `sqliteLibraryPath`
  - SQLite 动态库路径

### `realtime`

实时监测相关配置。

字段：

- `enabled`
  - 是否启用 realtime 配置段
- `telemetryTopic`
  - 实时遥测消费 topic
- `alarmTopic`
  - 实时告警消费 topic
- `statusTopic`
  - 实时状态消费 topic
- `maxLatestPoints`
  - latest 缓存上限建议值
- `trendBufferSize`
  - 趋势缓存点数
- `pushThrottleMs`
  - 推送节流时间

### `systemMonitor`

边端系统监测、远程诊断、配置拉取和共享内存点位快照配置。该服务用于平台侧查看网关本机运行情况，不用于第三方实时数据转发。

字段：

- `enabled`
  - 是否启用 `system-monitor@monitor-service.service`
- `defaultIntervalMs`
  - 没有页面订阅或订阅未指定周期时的默认上报周期
- `minIntervalMs`
  - 平台订阅允许的最小上报周期
- `subscriptionTtlSec`
  - 平台监测订阅租约有效期；页面不续约后边端停止高频上报
- `cpuAlertThreshold` / `memAlertThreshold` / `diskAlertThreshold`
  - CPU、内存、磁盘主动告警阈值
- `alertRepeatIntervalSec`
  - 同一告警持续存在时的重复上报间隔
- `diagEnabled`
  - 是否允许远程诊断命令
- `allowedCommands`
  - 允许执行的诊断命令白名单，当前包括 `uptime`、`free_m`、`df_root`、`top_once`、`ps_gateway`、`journal_tail`、`systemctl_status`、`cellular_status`
- `cellular.enabled`
  - 是否采集 4G 模块状态
- `cellular.probeIntervalMs`
  - 4G 状态实际探测间隔；即使页面订阅周期更短，也不会每次都访问串口
- `cellular.commandTimeoutMs`
  - 单条 AT 指令等待响应超时时间
- `cellular.atBaudRate`
  - 4G AT 口波特率，当前硬件默认 `115200`
- `cellular.signalAlertThresholdPercent`
  - 4G 信号强度低于该百分比时主动告警
- `cellular.maskSensitiveFields`
  - 是否脱敏 IMEI、ICCID、IMSI。当前项目要求平台侧直接查看完整 4G 模块标识，默认 `false`
- `cellular.interfacePatterns`
  - 4G 数据网卡匹配规则，例如 `wwan*`、`ppp*`、`usb*`、`rmnet*`
- `cellular.modemDevicePatterns`
  - 4G Modem 设备匹配规则。当前硬件优先按 `/dev/ttyUSB2`、`/dev/ttyUSB1`、`/dev/ttyUSB0`、`/dev/ttyUSB3` 顺序探测，之后兜底匹配 `/dev/ttyUSB*` 和 `/dev/cdc-wdm*`

### `localDisplay`

边端本地屏幕展示配置。该服务直接读取共享内存，通过本机 HTTP/SSE 页面展示点位，不依赖 MQTT Broker 和 Java 平台在线。Java 平台“本地画面”页面也读取同一段配置，并使用当前 `machineCode` 的实时 MQTT 缓存同步预览；平台预览不是 VNC 截屏。

字段：

- `enabled`
  - 是否启动 `local-display@monitor-service.service`
  - 出厂默认 `false`，需要边端本地屏幕时再开启
- `bindHost`
  - 本地监听地址；默认建议 `127.0.0.1`，只允许边端本机浏览器访问
- `port`
  - 本地展示端口，默认 `18080`
- `refreshIntervalMs`
  - 读取共享内存并推送页面的周期
- `maxPointsPerFrame`
  - 单帧最大点位数，防止浏览器一次渲染过多数据
- `showOnlyConfiguredPoints`
  - 是否只展示 `screens[].widgets[].bind` 中配置的点位；如果没有配置点位则自动回退到路由中的全部点位
- `sharedMemoryNames`
  - 本地展示服务需要打开的共享内存列表；为空时从 `mqttDriver` 和设备配置自动合并
- `kiosk.enabled`
  - 是否允许启动本机全屏展示进程 `local-kiosk@monitor-service.service`
- `kiosk.requireDisplayConnected`
  - 是否要求检测到显示输出后才启动本地画面
- `kiosk.requireEdid`
  - 是否要求 EDID 非空；默认建议 `true`
  - 如果硬件输出为 `DPI-1` 且没有 EDID，可显式配置为 `false`
- `kiosk.displayOutput`
  - 指定显示输出名，例如 `DPI-1`、`HDMI-1`
  - 留空时自动扫描 HDMI/DP/DPI/DSI/LVDS/VGA 等输出
- `kiosk.width` / `kiosk.height`
  - Kiosk 目标分辨率
- `kiosk.fullscreen`
  - 是否全屏
- `kiosk.setDisplayMode`
  - 是否尝试用 `xrandr` 切换显示模式
- `kiosk.zoom`
  - WebKit 缩放比例
- `screens`
  - WebKit 本地画面定义列表
- `screens[].screenCode`
  - 画面编码，例如 `overview`
- `screens[].title`
  - 画面标题
- `screens[].layout`
  - 画面栅格布局配置
- `screens[].layout.columns`
  - 栅格列数，默认 `12`
- `screens[].layout.rowHeight`
  - 栅格行高，单位像素
- `screens[].layout.gap`
  - 组件间距，单位像素
- `screens[].widgets`
  - 画面组件列表
- `screens[].widgets[].type`
  - 组件类型，支持 `groupTitle`、`statusCard`、`valueCard`、`pointTable`、`alarmSummary`
- `screens[].widgets[].bind.index`
  - 单点组件绑定的点位 index
- `screens[].widgets[].bind.indexes`
  - 多点组件绑定的点位 index 列表
- `screens[].widgets[].position`
  - 组件在栅格中的位置，字段为 `x`、`y`、`w`、`h`
- `screens[].widgets[].columns`
  - 表格类组件显示列
- `screens[].widgets[].valueFormat`
  - 单点值显示格式，支持 `number`、`raw`、`boolOnline`
- `viewTemplate`
  - 旧 HTML/CSS 模板兼容配置；当前生产默认关闭，推荐使用 `screens/widgets`
- `viewTemplate.refreshIntervalMs`
  - 旧模板兼容刷新周期。边端本机 `LocalDisplay` 仍按 `localDisplay.refreshIntervalMs` 刷新。

### `cameraService`

边端摄像头接入和推流配置。该服务独立运行，不依赖 Modbus/DLT645 采集驱动。视频流不写入共享内存，也不通过 MQTT 传输；摄像头在线状态、帧率、码率、错误码等轻量状态会写入共享内存，并通过 MQTT 状态 topic 上报。

字段：

- `enabled`
  - 是否允许启动 `camera-service@camera-service.service`
  - 出厂默认 `false`
  - `false` 时不启动边端摄像头服务，不注册摄像头共享内存状态点，Java 平台摄像头监控接口返回空 `streams` 且不查询 ZLMediaKit
  - `true` 时仍要求至少一个 `cameras[].enabled=true`，否则服务不启动
- `statusIntervalMs`
  - 摄像头状态写入共享内存和 MQTT 上报周期
- `sharedMemoryName`
  - 摄像头状态点写入的共享内存，默认 `gateway_point_store`
- `statusTopic`
  - 摄像头状态 MQTT topic 基础名，实际发送时会自动拼接 `/{machineCode}`
- `eventTopic`
  - 预留摄像头事件 topic 基础名
- `media.type`
  - 推流模式，当前实现为 `rtsp_push`
- `media.serverUrl`
  - 平台媒体服务 RTSP 地址。公网部署时填写平台固定公网 IP 或域名，例如 `rtsp://203.0.113.10:8554`
- `media.transport`
  - RTSP 拉流传输方式，默认 `tcp`
- `media.reconnectIntervalMs`
  - 推流进程异常退出后的重启间隔
- `media.auth.enabled`
  - 是否启用平台媒体入口认证
- `media.auth.mode`
  - 认证模式，当前支持 `basic` 和 `token_query`
- `media.auth.username` / `media.auth.password`
  - `basic` 模式使用；边端会自动生成 `rtsp://username:password@host:port/path`
- `media.auth.token`
  - `token_query` 模式使用
- `media.auth.tokenParam`
  - `token_query` 模式的 query 参数名，默认 `token`
- `media.auth.hideCredentialsInStatus`
  - 是否在 MQTT 状态上报和日志中隐藏推流凭据，生产建议 `true`
- `cameras`
  - 摄像头列表
- `cameras[].cameraCode`
  - 摄像头编码，同一网关内唯一
- `cameras[].enabled`
  - 是否启用该摄像头
- `cameras[].sourceType`
  - 摄像头来源，支持 `usb`、`uvc`、`rtsp`、`test`
- `cameras[].source`
  - 视频源地址；USB 示例 `/dev/video0`，RTSP 示例 `rtsp://192.168.1.50:554/stream1`
- `cameras[].sourceAuth`
  - 摄像头源认证配置，只用于边端访问现场局域网摄像头；字段和 `media.auth` 一致，常用 `basic` 用户名密码
- `cameras[].command`
  - 自定义推流命令；配置后优先使用该命令，适合特殊摄像头或 GStreamer 管线
- `cameras[].video.width` / `height` / `fps`
  - USB 或测试源编码参数
- `cameras[].video.codec`
  - `copy` 表示 RTSP 源直接转推；其他值默认用 H.264 编码推流
- `cameras[].video.bitrateKbps`
  - 编码目标码率
- `cameras[].stream.path`
  - 流路径，例如 `GW0001/CAM001`
- `cameras[].stream.publishUrl`
  - 完整推流地址；为空时由 `media.serverUrl + stream.path` 生成
- `cameras[].statusPointIndexes.online`
  - 摄像头在线状态点 index
- `cameras[].statusPointIndexes.fps`
  - 摄像头帧率状态点 index
- `cameras[].statusPointIndexes.bitrateKbps`
  - 摄像头码率状态点 index
- `cameras[].statusPointIndexes.errorCode`
  - 摄像头错误码状态点 index

### `ota`

OTA 升级执行配置。

字段：

- `enabled`
  - 是否启用 OTA 执行链
- `currentVersion`
  - 当前版本号
- `artifactBaseUrl`
  - 制品基地址
- `downloadDir`
  - 下载目录
- `stagingDir`
  - 暂存目录
- `backupDir`
  - 回滚备份目录
- `packageType`
  - 包格式，如 `tar.gz`
- `applyScript`
  - 升级执行脚本
- `rollbackScript`
  - 升级失败回滚脚本
- `checksumRequired`
  - 是否强制校验 `sha256`
- `autoReboot`
  - 是否自动重启，当前只保留配置
- `retentionCount`
  - 历史下载包保留数量，超出后会清理最旧文件
- `statusReportIntervalSec`
  - 状态上报节奏，当前阶段状态为即时推送
- `upgradeTimeoutSec`
  - 升级超时秒数，当前在下载后和校验后检查是否超时
- `downloadRetryCount`
  - OTA 下载最大尝试次数，默认 3
- `downloadRetryBackoffMs`
  - OTA 下载失败后的退避基准毫秒数，第 N 次重试等待 `downloadRetryBackoffMs * N`
- `storage.provider`
  - `local` 或 `minio`
- `storage.presignExpireMinutes`
  - 临时链接有效期
- `storage.minio.*`
  - MinIO 连接参数

### `config/samples/messages/mqtt_telemetry_chunk_example.json`

用于描述实时数据超过 `mqtt.maxPayloadBytes` 后的分片上报格式。

核心字段：

- `type`
- `machineCode`
- `chunked`
- `chunkId`
- `chunkIndex`
- `chunkCount`
- `meters[].meterCode`
- `meters[].values[]`

基础 topic：

```text
edge/telemetry
```

实际发布 topic：

```text
edge/telemetry/<machineCode>
```

处理规则：

- `chunkIndex` 从 `1` 开始。
- 同一批分片使用相同 `machineCode + chunkId`。
- 平台实时页面可以按片刷新，不必等待全量分片到齐。
- 如果业务需要完整快照，应收齐 `chunkCount` 片后再合并。
