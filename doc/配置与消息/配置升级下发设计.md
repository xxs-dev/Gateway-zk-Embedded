# 配置升级下发设计

## 1. 当前状态

当前配置升级链路已经收敛为 OTA 统一链路：

```text
平台或 Win 工具生成配置/程序包
  -> 上传或暴露 artifactUrl
  -> MQTT 发布 OTA request
  -> 边端 MqttDriver 接收请求
  -> 下载并校验制品
  -> 执行 ota-apply.sh
  -> 按 manifest 备份、替换、授权、daemon-reload、重启服务
  -> 发布 ota reply/status
  -> 失败时按 ota-rollback.sh 回滚
```

边端不再把配置升级和程序升级拆成两套机制。配置、二进制、systemd 文件、部署脚本都通过同一套 OTA 请求、状态、回滚链路处理。

## 2. 包类型

推荐在 `manifest.json` 中声明逻辑包类型：

- `config`：只更新配置文件。
- `binary`：只更新程序二进制。
- `full`：同时更新程序、配置、脚本和 systemd 文件。
- `factory`：出厂初始化包，通常由 Win 工具通过 SSH/SFTP 初始化使用，不作为普通 OTA 包直接下发。

文件扩展名仍由 `ota.packageType` 控制，当前常用为 `tar.gz`。

## 3. 推荐包结构

配置包示例：

```text
manifest.json
config/
  runtime/
    devices/
      device_slave_ttySP1.json
    apps/
      mqtt-service.json
      monitor-service.json
  templates/
    dlt645_2007_standard_points.json
```

完整包示例：

```text
manifest.json
bin/
  ModbusRtu
  Dlt645Driver
  DioDriver
  CanDriver
  IecDriver
  MqttDriver
  EventEngine
  ComputeEngine
  SystemMonitor
  pointctl
deploy/
  gateway-services.service
  gateway-services.sh
  gateway-run.sh
  ota-apply.sh
  ota-rollback.sh
config/
  runtime/
  templates/
```

## 4. Manifest 约定

每个需要应用到目标路径的文件必须在 `manifest.json` 的 `files[]` 中声明：

```json
{
  "packageType": "config",
  "version": "2026.06.26.001",
  "machineCode": "COMM202600102",
  "files": [
    {
      "path": "config/runtime/devices/device_slave_ttySP1.json",
      "target": "/opt/modbus-gateway/config/runtime/devices/device_slave_ttySP1.json",
      "sha256": ""
    },
    {
      "path": "deploy/ota-apply.sh",
      "target": "/opt/modbus-gateway/bin/ota-apply.sh",
      "mode": "0755",
      "sha256": ""
    },
    {
      "path": "deploy/gateway-services.service",
      "target": "/etc/systemd/system/gateway-services.service",
      "sha256": ""
    }
  ],
  "clean": [
    "/opt/modbus-gateway/config/runtime/devices",
    "/opt/modbus-gateway/config/runtime/apps"
  ],
  "restart": {
    "enabled": true,
    "services": [
      "gateway-services.service"
    ]
  },
  "rollback": {
    "enabled": true
  }
}
```

字段说明：

- `files[].path`：包内相对路径。
- `files[].target`：边端目标绝对路径，只允许写入脚本白名单内的配置、二进制和 systemd 路径。
- `files[].sha256`：可选；填写后脚本会校验文件内容。
- `files[].mode`：可选；用于脚本或二进制授权，例如 `0755`。
- `clean[]`：可选；只允许清理 `/opt/modbus-gateway/config/runtime/*` 中的受控目录。
- `restart.services[]`：可选；只允许重启白名单内的服务。

## 5. 边端 apply 行为

`ota-apply.sh` 当前行为：

1. 校验 `jobId/version/backupDir/stagingDir` 安全性。
2. 校验压缩包成员，禁止绝对路径、`..`、符号链接、设备文件和超大解压内容。
3. 解包到 `$stagingDir/$jobId`。
4. 读取 `manifest.json`。
5. 按 manifest 备份旧文件到 `$backupDir/$jobId`。
6. 按 manifest 复制新文件到目标路径。
7. 对脚本和二进制执行 `chmod`。
8. 如涉及 systemd 文件，执行 `systemctl daemon-reload`。
9. 按 `restart.services[]` 重启白名单服务。
10. 写入 `current_version.txt` 和 `upgrade_history.log`。

当前允许的关键目标路径包括：

- `/opt/modbus-gateway/config/runtime/devices`
- `/opt/modbus-gateway/config/runtime/apps`
- `/opt/modbus-gateway/config/runtime/logic`
- `/opt/modbus-gateway/config/runtime/tls`
- `/opt/modbus-gateway/bin/*.sh`
- `/opt/modbus-gateway/bin/<driver>`
- `/etc/systemd/system/*gateway*.service`
- `/etc/systemd/system/*driver*.service`

以脚本中的白名单为准，未列入白名单的路径会被拒绝，避免 OTA 包覆盖系统任意文件。

## 6. 边端 rollback 行为

`ota-rollback.sh` 当前行为：

1. 优先读取 `current_version.txt` 中的 `backupDir/workDir`。
2. 从备份目录恢复 `/opt/...` 和 `/etc/...` 下的旧文件。
3. 如果 apply 阶段生成了 `restart_services.txt`，按该文件重启服务。
4. 如果回滚涉及 TLS 或 systemd 文件，执行 `systemctl daemon-reload` 并确保 `gateway-services.service` 重启。
5. 写入 `rollback_<jobId>.txt` 和恢复清单。

## 7. MQTT OTA 请求

OTA 请求发布到 app 配置中的 `otaRequestTopic`，运行时会拼接 `machineCode`：

```text
edge/ota/request/<machineCode>
```

请求示例：

```json
{
  "jobId": "CONFIG_OTA_2026062610300001",
  "machineCode": "COMM202600102",
  "artifactUrl": "https://edge.example.com/packages/gateway-config-COMM202600102-2026.06.26.001.tar.gz",
  "version": "2026.06.26.001",
  "sha256": "...",
  "size": 12345,
  "upgradeMode": "download_install",
  "packageType": "config"
}
```

边端会发布：

- `otaReplyTopic/<machineCode>`：请求是否 accepted。
- `otaStatusTopic/<machineCode>`：下载、校验、应用、回滚等阶段状态。

## 8. 平台和 Win 工具分工

平台端：

- 维护基础数据、监测配置和项目配置。
- 生成配置 OTA 包并发布远端 MQTT OTA。
- 不负责现场选择驱动包版本和初始化设备。

Win 工具：

- 负责现场初始化、直连测试、驱动包选择、工程导入和配置发布。
- 可以通过 SSH/SFTP 初始化没有任何服务的新设备。
- 可以生成 `project` profile 初始化包，只安装项目需要的驱动。
- 可以把已拉取或本地编辑的工程重新打包并通过直连或 MQTT 下发。

## 9. 验证流程

最小验证步骤：

1. 生成包含 `manifest.json` 的配置包。
2. 计算并记录包 SHA256。
3. 发布 OTA 请求。
4. 订阅状态：

```sh
mosquitto_sub -h <broker> -t 'edge/ota/reply/+' -v
mosquitto_sub -h <broker> -t 'edge/ota/status/+' -v
```

5. 边端检查：

```sh
cat /opt/modbus-gateway/ota/staging/current_version.txt
tail -n 100 /opt/modbus-gateway/ota/staging/upgrade_history.log
systemctl status gateway-services.service --no-pager
/opt/modbus-gateway/bin/gateway-run.sh smoke
```

6. 如需验证回滚：

```sh
sh /opt/modbus-gateway/bin/ota-rollback.sh \
  /opt/modbus-gateway/ota/downloads/<package>.tar.gz \
  <version> \
  <jobId> \
  /opt/modbus-gateway/ota/backup \
  /opt/modbus-gateway/ota/staging
```

## 10. 关键约束

- MQTT 只传请求和状态，不直接传大文件。
- 边端只按 `artifactUrl` 下载包。
- 包必须有可追溯的 `version/jobId/sha256`。
- 配置、程序、脚本、systemd 更新都应走 manifest 白名单。
- 配置更新后推荐重启 `gateway-services.service`，由它扫描当前配置并只启动实际引用的驱动。
- 回滚验证必须覆盖配置文件、脚本、systemd 文件和服务重启。
