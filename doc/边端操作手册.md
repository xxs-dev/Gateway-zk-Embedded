# 边端网关操作手册

本文面向现场部署和运维，覆盖出厂初始化、运行配置、服务管理、MQTT/OTA/系统监测、共享内存查询和常见故障处理。默认安装目录为 `/opt/modbus-gateway`。

## 1. 目录约定

边端运行目录：

```text
/opt/modbus-gateway/
  bin/                         可执行文件和脚本
  config/runtime/device_identity.json  网关本机身份配置
  config/runtime/apps/          运行时应用配置
  config/runtime/devices/       运行时协议设备配置
  config/runtime/tls/           MQTT TLS CA、客户端证书和可选隧道兜底配置
  config/templates/             DLT645 标准点表等模板
  data/                         SQLite、MQTT 离线缓存、环形文件
  ota/                          OTA 下载、暂存、备份目录
  backup/                       安装或恢复前自动备份的配置
```

源码仓库中的出厂模板目录：

```text
config/factory/runtime/apps/mqtt-service.json
config/factory/runtime/apps/monitor-service.json
config/factory/runtime/device_identity.json
config/factory/runtime/devices/device_slave_ttySP1.json
config/factory/runtime/devices/device_slave_ttySP2.json
config/factory/runtime/devices/device_dio.json
```

## 2. 出厂默认配置

出厂模板默认启用两个 Modbus RTU 示例串口和本机 DI/DO。摄像头、本地画面默认关闭，避免无外设场景上电后占用资源。

| 文件 | 作用 |
| --- | --- |
| `device_identity.json` | 网关本机身份，包含 `machineCode`、`imei`、序列号、型号和版本信息 |
| `device_slave_ttySP1.json` | `/dev/ttySP1`，9600 8N1，从站 1，寄存器 0-9，在线状态点 `11000` |
| `device_slave_ttySP2.json` | `/dev/ttySP2`，9600 8N1，从站 1，寄存器 0-9，在线状态点 `12000` |
| `device_dio.json` | 本机 18 路 DI 和 8 路 DO，DI/DO 均按低有效处理，平台逻辑 `1=有效/闭合` |
| `mqtt-service.json` | 第三方转发服务，包含 MQTT 全量快照、告警/变位事件补发、OTA 接收 |
| `monitor-service.json` | 主站监测服务，包含资源监测、点查询、诊断命令、配置拉取 |
| `camera-service.json` | 摄像头推流服务，包含 USB/RTSP 摄像头托管、状态上报和共享内存状态点 |

出厂默认 `machineCode` 为 `GW_FACTORY_001`，正式上线前必须改成平台登记的唯一编号，例如 `GW0001`。

## 3. 首次安装

将交叉编译产物和仓库文件放到边端设备后，在仓库根目录执行：

```sh
sh deploy/install-factory-config.sh
```

推荐将出厂默认配置先打成一个独立包，再和初始化脚本一起放到 `/home`。这样现场初始化不依赖仓库目录结构：

```sh
sh deploy/build-factory-package.sh
scp gateway-factory-defaults.tar.gz root@<edge-ip>:/home/gateway-factory-defaults.tar.gz
scp deploy/install-factory-config.sh root@<edge-ip>:/home/install-factory-config.sh
ssh root@<edge-ip> "chmod +x /home/install-factory-config.sh && sh /home/install-factory-config.sh"
```

初始化脚本会按以下顺序寻找配置包：

```text
FACTORY_PACKAGE 指定路径
/home/gateway-factory/gateway-factory-defaults.tar.gz
/home/gateway-factory-defaults.tar.gz
仓库根目录 gateway-factory-defaults.tar.gz
/opt/modbus-gateway/gateway-factory-defaults.tar.gz
```

初始化包内容包含：

```text
gateway-factory-defaults/
  config/factory/runtime/
  config/templates/
  config/examples/
  config/README.md
  deploy/
  build-aarch64/              # 如果本机已完成交叉编译，则包含边端可执行文件
```

脚本会在交互终端提示填写以下初始化参数，并优先使用旧运行配置作为默认值：

| 参数 | 写入位置 |
| --- | --- |
| `runtimeMode` | `mqtt-service.json` 和 `monitor-service.json` 的 `runtimeMode`；默认 `gateway`，EMS 项目才填写 `ems` |
| `machineCode` | `config/runtime/device_identity.json` 的 `machineCode`，同时同步到所有 app 配置的 `mqtt.clientId` |
| `MQTT broker` | `mqtt-service.json` 和 `monitor-service.json` 的 `mqtt.broker` |
| `MQTT username/password` | `mqtt-service.json` 和 `monitor-service.json` 的 MQTT 登录信息 |
| `MQTT TLS caFile/certFile/keyFile` | 两个 app 配置的 `mqtt.tls.*` |
| `MQTT TLS insecureSkipVerify` | 仅调试自签证书时使用，生产建议保持 `false` |

非交互量产时可以用环境变量跳过提示：

```sh
FACTORY_PROMPT=0 \
INIT_RUNTIME_MODE=gateway \
INIT_MACHINE_CODE=GW0001 \
INIT_MQTT_BROKER=tcp://192.168.22.102:1883 \
INIT_MQTT_USERNAME=kyxn \
INIT_MQTT_PASSWORD='your-password' \
INIT_MQTT_TLS_ENABLED=false \
sh deploy/install-factory-config.sh
```

TLS broker 示例：

```sh
FACTORY_PROMPT=0 \
INIT_RUNTIME_MODE=gateway \
INIT_MACHINE_CODE=GW0001 \
INIT_MQTT_BROKER=ssl://broker.example.com:8883 \
INIT_MQTT_USERNAME=kyxn \
INIT_MQTT_PASSWORD='your-password' \
INIT_MQTT_TLS_ENABLED=true \
INIT_MQTT_CA_FILE=/opt/modbus-gateway/config/runtime/tls/ca.pem \
INIT_MQTT_CERT_FILE=/opt/modbus-gateway/config/runtime/tls/client.pem \
INIT_MQTT_KEY_FILE=/opt/modbus-gateway/config/runtime/tls/client.key \
INIT_MQTT_INSECURE_SKIP_VERIFY=false \
sh deploy/install-factory-config.sh
```

`gateway` 是生产初始化默认模式。该模式会从 app 配置中移除 EMS 本体虚拟设备引用和 `graphEms` 规则，平台端发布配置时也会跳过 EMS 逻辑配置步骤。只有储能 EMS 项目才显式指定：

```sh
INIT_RUNTIME_MODE=ems sh deploy/install-factory-config.sh
sh deploy/production-init.sh --runtime-mode ems
```

脚本会执行以下动作：

1. 创建 `/opt/modbus-gateway` 目录结构。
2. 安装 `build-aarch64` 下的可执行文件到 `/opt/modbus-gateway/bin`。
3. 安装 `deploy` 下的 systemd 服务模板和运行脚本。
4. 将 `config/factory/runtime` 复制为 `/opt/modbus-gateway/config/runtime`。
5. 如果原来已有运行配置，会备份到 `/opt/modbus-gateway/backup/runtime-YYYYMMDDHHMMSS`。
6. 将初始化参数同步写入 `config/runtime/apps/*.json`，避免 MQTT、监测和摄像头服务无法连接 broker。
7. 默认启动 `gateway-services.service`。

如果只安装不启动：

```sh
START_SERVICES=0 sh deploy/install-factory-config.sh
```

如果要清理旧共享内存后恢复出厂配置：

```sh
RESET_SHM=1 sh deploy/install-factory-config.sh
```

## 4. 上线前必须确认

如果首次安装时没有填写完整参数，仍可手工修改网关身份文件中的 `machineCode` 和 `imei`：

```sh
vi /opt/modbus-gateway/config/runtime/device_identity.json
```

设备串口/协议配置里不再维护 `machineCode`，只描述协议、串口、从站、电表、点表、共享内存和本地存储。

修改两个应用配置中的 MQTT 参数：

```text
/opt/modbus-gateway/config/runtime/apps/mqtt-service.json
/opt/modbus-gateway/config/runtime/apps/monitor-service.json
```

至少需要确认：

| 字段 | 说明 |
| --- | --- |
| `identityConfigFile` | 网关身份配置文件路径，默认 `/opt/modbus-gateway/config/runtime/device_identity.json` |
| `mqtt.broker` | MQTT Broker 地址，例如 `tcp://192.168.22.102:1883` |
| `mqtt.protocolVersion` | `mqtt3` 或 `mqtt5` |
| `mqtt.clientId` | 配置值保持为 `machineCode`；边端子服务和平台端消费者会在运行时自动追加职责后缀，避免 MQTT clientId 冲突 |
| `mqtt.username` / `mqtt.password` | MQTT 登录信息 |
| `deviceConfigFiles` | 当前应用需要读取的设备配置文件 |
| `mqttDriver.sharedMemoryNames` | MQTT 转发服务需要扫描的共享内存列表 |

配置改完后重启：

```sh
/opt/modbus-gateway/bin/gateway-run.sh restart
```

## 5. 服务模型

`gateway-services.service` 是总入口，实际启动哪些服务由配置文件决定：

| 配置存在或开关 | 启动服务 |
| --- | --- |
| app 配置 `deviceConfigFiles[]` 引用的设备 JSON 且 `protocol.type=modbus_rtu/modbus_tcp` | `modbus-rtu@<配置文件名>.service` |
| app 配置 `deviceConfigFiles[]` 引用的设备 JSON 且 `protocol.type=dlt645_2007` | `dlt645-driver@<配置文件名>.service` |
| app 配置 `deviceConfigFiles[]` 引用的设备 JSON 且 `protocol.type=local_dio` | `dio-driver@<配置文件名>.service` |
| `mqtt-service.json` 存在 | `mqtt-driver@mqtt-service.service` |
| `mqtt-service.json:eventEngine.enabled=true` | `event-engine@mqtt-service.service` |
| `monitor-service.json:localDisplay.enabled=true` | `local-display@monitor-service.service` |
| `monitor-service.json:localDisplay.enabled=true` 且 `localDisplay.kiosk.enabled=true`、显示输出检测通过 | `local-kiosk@monitor-service.service` |
| `monitor-service.json:systemMonitor.enabled=true` | `system-monitor@monitor-service.service` |
| `camera-service.json:cameraService.enabled=true` 且存在启用摄像头 | `camera-service@camera-service.service` |

未被 `deviceConfigFiles[]` 引用的设备 JSON 不会启动。没有某类协议配置时，对应协议驱动不会运行。

出厂默认 `gateway-run.sh list` 应至少包含：

```text
modbus-rtu@device_slave_ttySP1.service
modbus-rtu@device_slave_ttySP2.service
dio-driver@device_dio.service
event-engine@mqtt-service.service
mqtt-driver@mqtt-service.service
system-monitor@monitor-service.service
```

摄像头服务的总开关更严格：`cameraService.enabled=false` 时不启动 `camera-service@camera-service.service`，手工启动 `CameraService` 也会直接退出；同时不会注册摄像头共享内存状态点。只有总开关打开并且至少一个摄像头 `enabled=true`，摄像头服务才属于运行态。

本地屏幕展示：

```sh
systemctl status local-display@monitor-service.service --no-pager
systemctl status local-kiosk@monitor-service.service --no-pager
wget -qO- http://127.0.0.1:18080/api/health
```

出厂默认 `localDisplay.enabled=false`，不会主动启动本地画面。需要本地屏幕时，在 `monitor-service.json` 中打开 `localDisplay.enabled=true`。

如果边端接了显示器，且显示检测通过，则通过 `local-kiosk@monitor-service.service` 全屏展示。也可以在边端本机浏览器打开：

```text
http://127.0.0.1:18080/
```

主平台“本地画面”页面使用同一份 `localDisplay` 配置和当前 `machineCode` 的实时 MQTT 缓存进行同步预览。平台预览不是 VNC 截屏；边端本地屏幕仍然直接读共享内存，不依赖平台在线。

摄像头服务查看：

```sh
systemctl status camera-service@camera-service.service --no-pager
journalctl -u camera-service@camera-service.service -n 100 --no-pager
```

出厂默认 `cameraService.enabled=false`，不会主动启动摄像头推流。启用后需要确保边端存在 `ffmpeg`，并且平台侧 RTSP/媒体服务地址可达。

本机 DI/DO 查看：

```sh
systemctl status dio-driver@device_dio.service --no-pager
journalctl -u dio-driver@device_dio.service -n 100 --no-pager
/opt/modbus-gateway/bin/pointctl get --index 410006 --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json
/opt/modbus-gateway/bin/pointctl write --index 420001 --value 1 --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json
/opt/modbus-gateway/bin/pointctl write --index 420001 --value 0 --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json
```

DIDO 默认规则：

- `DI1-DI18`：低有效，GPIO 低电平对应平台值 `1=有效`。
- `DO1-DO8`：低有效，平台写 `1` 时继电器闭合，平台写 `0` 时继电器断开。

显示检测说明：

- 默认要求 DRM connector `connected` 且 EDID 非空。
- 如果板卡通过 `DPI-1` 输出且 EDID 始终为空，需要配置 `localDisplay.kiosk.displayOutput="DPI-1"` 和 `localDisplay.kiosk.requireEdid=false`。
- `local-kiosk@.service` 有 `MemoryHigh=384M`、`MemoryMax=512M`，防止 WebKit 长时间运行导致内存占用失控。

## 6. 常用运行命令

查看配置会启动哪些服务：

```sh
/opt/modbus-gateway/bin/gateway-run.sh list
```

启动、停止、重启：

```sh
/opt/modbus-gateway/bin/gateway-run.sh start
/opt/modbus-gateway/bin/gateway-run.sh stop
/opt/modbus-gateway/bin/gateway-run.sh restart
```

查看状态：

```sh
/opt/modbus-gateway/bin/gateway-run.sh status
```

跟踪日志：

```sh
/opt/modbus-gateway/bin/gateway-run.sh logs
```

查看共享内存点表：

```sh
/opt/modbus-gateway/bin/gateway-run.sh snapshot
```

查看待写队列：

```sh
/opt/modbus-gateway/bin/gateway-run.sh pending
```

查看共享内存容量、占用和写队列：

```sh
/opt/modbus-gateway/bin/gateway-run.sh stats
```

执行生产健康检查：

```sh
/opt/modbus-gateway/bin/gateway-run.sh health
```

执行生产冒烟检查：

```sh
/opt/modbus-gateway/bin/gateway-run.sh smoke
```

`smoke` 覆盖身份、配置引用、TLS、服务、共享内存、磁盘、OTA 脚本和关键安全项。最终必须 `fail=0` 才允许进入生产；详细说明见 [生产验收冒烟测试.md](生产验收冒烟测试.md)。

确认要清理压测或异常退出留下的运行缓存时：

```sh
/opt/modbus-gateway/bin/gateway-run.sh cleanup
```

`cleanup` 会停止服务、删除 `/dev/shm/gateway_point_store*` 和 `/tmp/gateway-stress-*` / `/tmp/gateway-ci-*`。生产现场只能在停服务维护窗口执行。

## 7. 点位与共享内存

生产默认所有接口和协议使用同一个共享内存 `gateway_point_store`，平台、MQTT、事件引擎和诊断工具都从这个共享内存统一取数。多共享内存仍可作为后续隔离扩展，但出厂模板不再默认按串口拆分。

出厂默认：

| 串口 | 共享内存 | SQLite |
| --- | --- | --- |
| `/dev/ttySP1` | `gateway_point_store` | `/opt/modbus-gateway/data/point_samples_ttySP1.db` |
| `/dev/ttySP2` | `gateway_point_store` | `/opt/modbus-gateway/data/point_samples_ttySP2.db` |

点位关键字段：

| 字段 | 说明 |
| --- | --- |
| `index` | 全局唯一点位编号，同一网关内不能重复 |
| `machineCode` | 网关唯一编号，来自 `device_identity.json`，作为平台侧主分组 |
| `meterCode` | 表计、从站或设备编号 |
| `pointCode` | 点位业务编码 |
| `fullUpload` | 是否进入 MQTT 全量快照上传 |
| `reportOnChange` | 是否产生变位事件 |
| `read.dataType=device_online` | 设备在线状态点，不读取寄存器，由采集结果自动写入 |

## 8. MQTT 主题

配置文件中只填写基础 topic，边端运行时按 `machineCode` 追加后缀，最终 topic 规则固定为：

```text
<baseTopic>/<machineCode>
```

例如 `machineCode=GW0001` 时，基础 topic `edge/telemetry` 的实际发布 topic 是 `edge/telemetry/GW0001`。如果配置里已经写成 `edge/telemetry/GW0001`，边端不会重复追加。

`machineCode` 来源于 `identityConfigFile` 指向的 `device_identity.json`。如果旧设备 JSON 里仍保留 `machineCode`，它必须和身份文件一致，否则进程会拒绝启动，避免同一网关身份冲突。

常用 topic：

| 业务 | 配置中的基础 topic | 实际 topic 示例 |
| --- | --- | --- |
| 全量快照 | `edge/telemetry` | `edge/telemetry/GW0001` |
| 变位事件 | `edge/event/change` | `edge/event/change/GW0001` |
| 告警事件 | `edge/alarm` | `edge/alarm/GW0001` |
| 状态事件 | `edge/status` | `edge/status/GW0001` |
| 控制下发 | `edge/command/request` | `edge/command/request/GW0001` |
| 控制回执 | `edge/command/reply` | `edge/command/reply/GW0001` |
| OTA 下发 | `edge/ota/request` | `edge/ota/request/GW0001` |
| OTA 回执 | `edge/ota/reply` | `edge/ota/reply/GW0001` |
| OTA 状态 | `edge/ota/status` | `edge/ota/status/GW0001` |
| 系统监测请求 | `edge/system/monitor/request` | `edge/system/monitor/request/GW0001` |
| 系统监测回执 | `edge/system/monitor/reply` | `edge/system/monitor/reply/GW0001` |
| 系统资源上报 | `edge/system/monitor/telemetry` | `edge/system/monitor/telemetry/GW0001` |
| 系统资源告警 | `edge/system/monitor/alert` | `edge/system/monitor/alert/GW0001` |
| 共享内存点查询 | `edge/system/monitor/points` | `edge/system/monitor/points/GW0001` |
| 诊断命令请求 | `edge/system/diag/request` | `edge/system/diag/request/GW0001` |
| 诊断命令回执 | `edge/system/diag/reply` | `edge/system/diag/reply/GW0001` |
| 配置拉取请求 | `edge/config/pull/request` | `edge/config/pull/request/GW0001` |
| 配置拉取回传 | `edge/config/pull/reply` | `edge/config/pull/reply/GW0001` |

平台侧推荐按 `edge/telemetry/+`、`edge/alarm/+` 这种通配订阅发现网关，再按具体 `edge/telemetry/GW0001` 精确处理单台网关。下行命令、OTA、系统监测和配置拉取必须发布到带 `machineCode` 后缀的实际 topic。

## 9. MQTT TLS 生产接入

内置 MQTT Publisher 已支持 TLS，不需要额外 MQTT SDK。生产跨公网或不可信网络时，将 `mqtt-service.json` 和 `monitor-service.json` 中的 broker 与 TLS 配置改为：

```json
"broker": "ssl://broker.example.com:8883",
"tls": {
  "enabled": true,
  "caFile": "/opt/modbus-gateway/config/runtime/tls/ca.crt",
  "certFile": "",
  "keyFile": "",
  "insecureSkipVerify": false
}
```

边端需要存在 OpenSSL 运行库 `libssl` / `libcrypto`。如果 broker 要求双向 TLS，`certFile` 和 `keyFile` 必须成对填写。`insecureSkipVerify=true` 只允许临时联调，生产禁止开启。

上线前检查：

```sh
ldconfig -p | grep -E 'libssl|libcrypto'
test -f /opt/modbus-gateway/config/runtime/tls/ca.crt
```

如果目标系统无法提供 OpenSSL 运行库，可退回 `mqtt-tls-tunnel@.service` 本机 stunnel 方案：app broker 写 `tcp://127.0.0.1:18883`，TLS 参数放在 `config/runtime/tls/mqtt-service-stunnel.conf`。`gateway-services.service` 会在 broker 指向本机 stunnel 监听端口且配置文件存在时自动启动隧道实例。

## 10. OTA 配置下发

边端由 `mqtt-service.json` 负责 OTA：

```json
"ota": {
  "enabled": true,
  "downloadDir": "/opt/modbus-gateway/ota/downloads",
  "stagingDir": "/opt/modbus-gateway/ota/staging",
  "backupDir": "/opt/modbus-gateway/ota/backup",
  "applyScript": "/opt/modbus-gateway/bin/ota-apply.sh",
  "rollbackScript": "/opt/modbus-gateway/bin/ota-rollback.sh",
  "retentionCount": 3
}
```

平台生成配置包后发布到 OTA topic，边端流程为：

1. 下载 OTA 包，支持 HTTP/HTTPS。
2. 校验 `sha256`。
3. 解压到 staging。
4. 执行 `ota-apply.sh`。
5. 重启需要变化的服务。
6. 写入 OTA 状态断点，MQTT 重连后继续上报最终状态。

`device_identity.json` 默认会被边端保留，不会被配置 OTA 包覆盖。只有 manifest 显式声明 `updateIdentity=true` 时，才允许 OTA 更新网关身份，避免批量下发时把多台设备改成同一个 `machineCode` 或 `imei`。

如果升级失败，会执行 `ota-rollback.sh` 并发布失败状态。

## 11. 系统监测与诊断

系统监测由 `monitor-service.json` 和 `system-monitor@monitor-service.service` 负责，和第三方转发用的 `mqtt-service` 分离。

启用开关：

```json
"systemMonitor": {
  "enabled": true,
  "defaultIntervalMs": 5000,
  "cpuAlertThreshold": 90,
  "memAlertThreshold": 90,
  "diskAlertThreshold": 90
}
```

主站打开监测页面后，平台下发订阅请求，边端按订阅周期上报 CPU、内存、磁盘、网络和进程信息。即使平台未打开监测页面，当 CPU、内存或磁盘超过阈值时也会主动上报告警。

允许的诊断命令在 `allowedCommands` 中配置。边端只执行白名单命令，不执行任意 shell 字符串。

## 12. 配置拉取

平台可通过 MQTT 配置拉取请求让边端回传运行配置。大文件会分片回传，每片包含字节偏移、总字节数、片序号和总片数，平台按 `requestId + filePath` 重组。

用于现场排查的典型流程：

1. 平台选择 `machineCode`。
2. 下发配置拉取请求。
3. 边端分片回传 `runtime/device_identity.json`、`runtime/apps` 和 `runtime/devices`。
4. 平台显示当前快照，并与本地配置进行差异对比。
5. 修改后生成 OTA 配置包下发。

## 13. 故障处理

查看哪些服务应该启动：

```sh
/opt/modbus-gateway/bin/gateway-run.sh list
```

`gateway-run.sh list` 只读取 app 配置中的 `deviceConfigFiles[]`。`runtime/devices` 目录下存在但未被 app 配置引用的 JSON 不会启动，避免样例文件误拉起驱动。

实际服务没启动：

```sh
systemctl status gateway-services.service --no-pager
journalctl -u gateway-services.service -n 100 --no-pager
```

Modbus 无响应：

```sh
journalctl -u 'modbus-rtu@*.service' -f
```

检查串口配置：

```sh
grep -R -E '"serialPort"|"baudRate"|"parity"' /opt/modbus-gateway/config/runtime/devices
```

MQTT 无法连接：

```sh
journalctl -u mqtt-driver@mqtt-service.service -n 100 --no-pager
grep -R -E '"broker"|"clientId"|"username"' /opt/modbus-gateway/config/runtime/apps
```

共享内存异常或点位重复：

```sh
/opt/modbus-gateway/bin/pointctl snapshot --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json
/opt/modbus-gateway/bin/pointctl stats --app-config /opt/modbus-gateway/config/runtime/apps/mqtt-service.json
```

如果两个正在运行的驱动声明了相同 `index`，后启动的进程会在注册点位时失败，日志包含 `point.index already claimed by another active process`。

如果确认需要清空旧共享内存，先停服务再删除：

```sh
/opt/modbus-gateway/bin/gateway-run.sh cleanup
/opt/modbus-gateway/bin/gateway-run.sh start
```

## 14. 扩展新接口

新增 485 口的推荐方式：

1. 复制 `device_slave_ttySP1.json` 为 `device_slave_ttySPx.json`。
2. 修改 `protocol.transport.serialPort`。
3. 保持 `memoryStore.sharedMemoryName=gateway_point_store`，除非明确要做共享内存隔离。
4. 修改 `memoryStore.sqlitePath`。
5. 保证所有 `index` 全局唯一。
6. 不要在设备配置里新增 `machineCode`，网关身份只从 `device_identity.json` 获取。
7. 将新文件路径加入 `mqtt-service.json` 和 `monitor-service.json` 的 `deviceConfigFiles`。
8. 默认不需要修改 `mqttDriver.sharedMemoryNames`，保持只包含 `gateway_point_store`。
9. 重启 `gateway-services.service`。

同一物理串口只能运行一个协议驱动，不要在同一个串口上同时配置 Modbus 和 DLT645。
